> Total: 3
> Critical: 0 | High: 2 | Medium: 1 | Low: 0

## 1. Stop committing (and fsyncing) the whole index once per finished job

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: index-write / lock contention
- **File**: crates/engine-search/src/lib.rs:122-164 (`write_then_commit`, `index`) — caller: `crates/server/src/worker.rs:231-235`
- **Scenario**: The worker calls `state.search.index(docs)` once per successfully completed job, and `index()` unconditionally ends in `w.commit()` + `reader.reload()`. A crawl or fleet burst produces hundreds of small jobs; each one pays a full Tantivy commit — segment flush + fsync of the memory-mapped directory + a reader reload — for as little as one document. Worse, every mutating path shares `Arc<Mutex<IndexWriter>>` and holds that lock *across* the commit (line 129-132), so N concurrent workers do not just contend for the writer, they serialize behind each other's fsync. `delete_ids` / `delete_dataset` queue behind the same lock.
- **Root cause**: Commit granularity is bound to the caller's batch boundary rather than to a durability/freshness policy. Tantivy is designed for amortized commits (one segment per commit; the default merge policy then has to merge the resulting confetti). The `write_then_commit` helper — correct as a dedup of the epilogue — hard-wires commit into every write.
- **Impact**: Commit dominates indexing cost at small batch sizes (tens of ms of fsync + segment write vs. sub-ms `add_document`), so a 200-job burst pays ~200 fsyncs where ~5 would do, plus 200 tiny segments feeding sustained background merge I/O. Concurrent workers see the commit latency serialized, not overlapped. Metrics to watch: time inside `write_then_commit` (histogram), segment count in `cfg.dir` after a burst, and job-completion p95 with/without the change.
- **Fix sketch**: Split the epilogue: keep the lock → `edit` step, but move `commit()` behind a policy — a single background committer task (tokio interval, e.g. ≤250ms or ≥N pending docs) that locks the writer, commits, reloads, and notifies waiters; or expose `flush()` on the trait and let callers batch. Constraint to respect: the saved-search runner (`worker.rs:408-423`) queries immediately after `index()` and needs those docs visible, so it must await the next commit ticket (a `tokio::sync::watch` of commit generation, or an explicit `flush()` call on that one path) rather than assume synchronous visibility. Trade-off: search freshness lags by up to the commit interval; docs already state the index is a derived artifact, so a bounded lag is acceptable — silent loss of the uncommitted tail on hard kill is the real cost and is why the interval must stay small.

## 2. Add a backfill/reindex path — a wiped or late-enabled index only refills as jobs happen to re-run

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap / operability
- **File**: crates/engine-search/src/lib.rs:64-80 (`TantivyIndex::new` schema-drift rebuild), 140-164 (`Search::index` is the only write-in path)
- **Scenario**: Three ordinary situations leave the index silently wrong and unfixable: (a) `TantivyIndex::new` detects a pre-snippet schema and does `remove_dir_all` → recreate empty (line 69-77) — every previously indexed doc is gone; (b) a user runs with `[search] enabled=false` (`NoSearch`) for a while, then turns it on — everything scraped in that window is unsearchable; (c) the index directory is lost/corrupted or a schema change lands in future. In all three cases the only refill mechanism is the worker's post-job `index()` call, so a dataset only becomes searchable again when its app *next runs* — days for a weekly schedule, never for a retired app. The stored records are still in the DB, and `worker::dataset_search_docs` already knows how to turn live dataset records into `SearchDoc`s, so the data to rebuild from exists; nothing is wired to walk it. There is also no way to *observe* the problem: `Search` exposes no doc count, `GET /search` returns hits and facets only, and the rebuild is a `tracing::warn!` the operator likely never sees. (`crates/server/src/bin/reindex.rs` is simhash-only — it does not touch the search index.)
- **Root cause**: The index is treated as a write-through side effect of the job pipeline; "derived artifact" was used to justify wiping it (line 70-71 comment) without building the derivation. The `Search` trait has index/query/delete but no rebuild and no introspection.
- **Impact**: Product-visible silent data loss on the search surface — the search box and every saved-search alert quietly under-report against an index that looks healthy (queries return 200 with fewer hits). Affects the grants `unified` namespace hardest, where per-record docs are the whole value.
- **Fix sketch**: Two small pieces. (1) Add `async fn doc_count(&self) -> Result<u64>` to `Search` (`TantivyIndex` gets it from `reader.searcher().num_docs()`; `NoSearch` returns 0) and surface it on a status/health route so an emptied index is visible. (2) Add a rebuild entry point — a `--bin search-backfill` (mirroring `bin/reindex.rs`) or `POST /search/reindex?app=&dataset=` that streams live dataset records through the existing `dataset_doc` builder into `Search::index` in chunks; ids are already stable (`"<app>:<dataset>:<key>"`) so it is idempotent and upserts rather than duplicates. Have the schema-drift branch in `new()` log a loud, actionable message pointing at that command instead of implying self-healing.

## 3. Index a timestamp so results can be sorted and filtered by recency

- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/engine-search/src/lib.rs:23-31 (`Fields`), 52-61 (schema), 255-257 (`TopDocs::with_limit(...).order_by_score()`)
- **Scenario**: The schema has exactly six fields — id, app, dataset, url, title, body — no time dimension anywhere, and `SearchDoc` (`core/src/search.rs:16-23`) carries none either. So on a platform whose entire purpose is scraping the same sources repeatedly, a user cannot ask the two questions that matter most on a changing corpus: "newest first" and "only what appeared in the last 7 days". A grants search for `"AI research"` ranks a closed 2024 opportunity above one posted yesterday whenever BM25 says so. The same gap hits saved-search alerts, which currently re-run the whole query after every job and lean entirely on the `saved_search_seen` claim table to avoid re-alerting.
- **Root cause**: The index was built as a pure relevance surface; recency lives in the SQLite dataset rows (`fetched_at`/upsert time), never crosses into the Tantivy doc, so it is not available to rank or filter on at query time.
- **Impact**: Bounded but broad — every query on a corpus with any history is ranked by relevance alone. Unblocks recency sort, date-range scoping, "what's new" feeds, and lets alert queries be genuinely time-scoped rather than seen-table-scoped. Note this is a schema change: it trips the `body_is_stored` style drift check, so it should land together with finding #2's backfill (which is precisely what makes a schema change cheap).
- **Fix sketch**: Add `indexed_at: i64` to `SearchDoc` (populated by `worker::dataset_doc` / `record_doc` from the record's stored timestamp, falling back to now) and a `FAST | INDEXED | STORED` date/i64 field to the schema in `TantivyIndex::new`. Extend `SearchRequest` with `sort: Option<SearchSort>` (`Score` | `Newest`) and `since: Option<i64>`: `Newest` swaps `order_by_score()` for `order_by_fast_field` on the new field; `since` pushes a `RangeQuery` clause into the existing `clauses` vec next to the app/dataset `TermQuery` filters (line 229-248). Generalize `body_is_stored` into a schema-version/fingerprint check so this and future field additions trigger the rebuild path deliberately rather than by ad-hoc probe.
