## 1. Collapse concurrent stale acquires onto one Chrome launch per profile

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: hot-path / thundering-herd
- **File**: crates/engine-browser/src/lib.rs:201-237
- **Scenario**: `acquire()` now correctly launches **outside** the `holders` lock (the 2026-07-14 fix). But the fast path at :209-218 returns early **only when the holder is fresh**. When it is stale — cold start, crash (`alive == false`), or the `recycle_after_renders` boundary — every concurrent caller falls through to `self.launch(&user_data_dir).await` at :223. With the default `max_concurrent_renders = 4`, a crash or a recycle tick makes up to 4 tasks each spawn a full Chrome against the **same** `user_data_dir`. The re-check under the lock at :228-235 then keeps one and drops the other three. The prior fix sketch called for a placeholder so same-profile acquires await one launch; that half was never implemented, and moving the launch out of the lock is exactly what turned a serialized launch into a racing one.
- **Root cause**: staleness is evaluated per-caller with no in-flight-launch reservation for the key. Nothing between the fast-path miss and the re-lock marks "a launch for this key is already running", so the check-then-launch window is wide open to every task that misses.
- **Impact**: Per stale event: N−1 wasted Chrome launches (~up to chromiumoxide's 20 s `launch_timeout` each) and an N× transient Chrome memory spike (each instance is budgeted `--js-flags=--max-old-space-size=512`, so a 4-way race can transiently reserve ~2 GB of V8 headroom for work that is immediately discarded). The waste is amplified because `render()` holds its `render_slots` permit across `acquire()` (:259-269), so all 4 semaphore slots sit blocked on duplicate launches while renders that would have hit a live browser queue behind them — throughput drops to ~0 for the launch window. Separately, this is a correctness hazard, not just waste: Chromium enforces a single-instance lock per `--user-data-dir`, and duplicate launches target the identical dir. The project already knows this — `tests/render.rs:1-3` gives each test its own profile dir precisely so they "don't contend for the single-instance user-data lock when run in parallel". The recycle path overlaps even without concurrency: the old `LiveBrowser` is only dropped at the `holders.live.insert` on :234, i.e. **after** the replacement Chrome has launched, so a relaunch always races the still-live instance for the same dir. Suggested metrics: count launches vs. distinct `(key, stale-event)` pairs (ratio should be 1.0); histogram `launch` wall time; gauge resident Chrome count vs. `MAX_LIVE_PROFILES`.
- **Fix sketch**: Reserve the key under the lock before releasing it. Store `HashMap<String, Shared<BoxFuture<Result<Arc<ChromeBrowser>>>>>` (or a `tokio::sync::OnceCell` per key) alongside `Holders.live`: on a stale/missing key, insert the shared launch future under the lock, drop the lock, `await` the clone — so the 2nd..Nth caller awaits the *same* launch instead of starting its own, and other profiles still proceed unblocked. For the recycle/crash path specifically, `holders.live.remove(&key)` **before** launching so the outgoing Chrome releases its `user_data_dir` singleton lock first (in-flight renders keep their `Arc<ChromeBrowser>` clone from `checkout`, so dropping the map entry does not kill a render mid-flight). Also wrap `launch` in an explicit `tokio::time::timeout` well under 20 s. `is_stale` stays pure and its unit tests unchanged.

## 2. Cap the browser tier's captured HTML the way the HTTP tier caps bodies

- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: allocation / resource-bound
- **File**: crates/engine-browser/src/lib.rs:376-385
- **Scenario**: The HTTP tier streams its body and aborts the instant the cumulative size would pass a cap — `read_body_capped` (engine-http/src/lib.rs:455-466) enforcing `HttpRequest.max_body_bytes` else `[http] max_body_bytes` (16 MiB), and `fetching.md` sells this as the guard so "one huge/hostile URL can't balloon memory". The browser tier has no equivalent. `page.content().await` at :376 materializes the entire serialized DOM into an unbounded `String`, and `req.evaluate`'s result is deserialized into an unbounded `serde_json::Value` at :363-372. A `strategy = "browser"` fetch — or any `auto` fetch that escalates past a thin/blocked HTTP tier onto exactly the heavy JS-built pages that grow huge DOMs — bypasses the memory guard entirely. So the cap protects the cheap tier and not the expensive one.
- **Root cause**: The size cap was implemented as a property of reqwest body streaming rather than as a tier-agnostic contract on fetched content. `RenderRequest` (core/src/engine.rs:180-202) has no `max_body_bytes` field, so there is no knob to enforce even if the engine wanted one.
- **Impact**: Bounded but real. Chrome's own `--js-flags=--max-old-space-size=512` caps V8 heap, not DOM/serialized-HTML size, so a pathological page can hand back tens of MB; the string is then copied again into `RenderedPage.html` and onward through markdown/text extraction. At `max_concurrent_renders = 4` that is 4 concurrent uncapped buffers plus their downstream copies, with no typed error to attribute the spike to a URL. The HTTP tier's identical scenario is a clean `Error::Http` naming the cap and the URL.
- **Fix sketch**: Add `max_body_bytes: Option<u64>` to `RenderRequest` (serde-defaulted `None` = current behavior) and a `[browser] max_html_bytes` config default mirroring `[http] max_body_bytes`. After `page.content()`, check `html.len()` against the effective cap and return the same shape of typed error `Error::Browser` naming cap + URL, so both tiers fail identically on a hostile page. Cheaper and stricter, if worth the CDP work: `Runtime.evaluate` `document.documentElement.outerHTML.length` before capturing, and refuse over-cap pages without ever serializing them across the protocol. Reuse the pure `would_exceed_cap` helper's testing pattern so the decision is unit-testable without Chrome.

## 3. Add scripted page actions (scroll / click / wait) to the browser tier

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/engine-browser/src/lib.rs:322-372 (RenderRequest at crates/core/src/engine.rs:180-202)
- **Scenario**: A user points pumper at an infinite-scroll listing (job boards, marketplaces, most modern SPA search results) or a page behind a "Load more" button. The browser tier renders exactly one viewport of the first paint and returns. There is no way to express "scroll to the bottom 5 times, waiting for new rows each time, then capture" — so the product silently returns the first 20 of 500 records, and the dataset looks healthy. This is the single biggest class of pages the browser tier exists for and cannot actually reach.
- **Root cause**: `RenderRequest` exposes a fixed, one-shot lifecycle: `goto` → `wait_for_navigation` → optional `wait_for_selector` poll → `extra_wait_ms` settle → optional single `evaluate` → `content()`. The `evaluate` escape hatch looks like it could drive interaction, but the ordering defeats it: `evaluate` runs at :363-372 and `page.content()` at :376 with **no wait in between**, so a JS `window.scrollTo(...)` or `.click()` returns immediately and the DOM is captured before the lazy-loaded content lands. The gap exists because the engine was built for the static "render a JS page once" case; the vision backlog covers adjacent surfaces (#156 screenshot/PDF, #103 auto-follow pagination in the *extractor* via link-following, #122 extraction-by-demonstration) but none address in-page JS pagination inside a render.
- **Impact**: Product-limiting. Infinite-scroll/lazy-load is the dominant pattern for the JS-heavy pages that trigger escalation to the browser tier in the first place, so the expensive tier under-delivers on precisely its target workload — and it under-delivers **silently** (a truncated listing is not thin content, so it passes `min_content_chars`, wins the tier, and never escalates). Unlocks whole site categories with no new engine.
- **Fix sketch**: Add `actions: Vec<PageAction>` to `RenderRequest` (serde-defaulted empty = exactly current behavior), executed after the settle sleep and before the existing `evaluate`. A small enum covers the real cases: `Scroll { to: Bottom | Pixels(i64) }`, `Click { selector: String }`, `Type { selector: String, text: String }`, `WaitFor { selector: String, timeout_ms: Option<u64> }`, `WaitMs(u64)`, and `Repeat { times: u32, until_selector_count_stable: bool, steps: Vec<PageAction> }` for the scroll-until-exhausted loop. Each maps onto machinery already in this file — `page.evaluate` for scrolls, `page.find_element` for clicks, and the deadline-poll loop at :341-356 generalized into a reusable `wait_for(selector, deadline)` helper shared by `wait_for_selector` and `WaitFor`. Report what happened in `RenderedPage` (e.g. `actions_completed: usize`, mirroring the existing honest `nav_timed_out` / `selector_found` / `blocked_resources` signals) so a truncated listing is visible rather than silent. Thread the field through `FetchRequest` for parity with `profile`, and cap total action time against `nav_timeout_secs` so a `Repeat` can't run forever.
