## 1. Expose a metered engine seam so the crawler stops bypassing cost + tier learning

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/core/src/app.rs:44-60, 114-227
- **Scenario**: An operator runs the `crawl` app over 10k pages, then opens the cost/ROI overview (`CostLedger::summary`) to see what the crawl cost and which hosts need the browser tier. The crawl reports **$0 and zero engine events**, and tier memory learned nothing from 10k real fetches — while a single-page `readable` job (1 fetch) is fully attributed.
- **Root cause**: `AppContext::fetch` is the only metered seam, and it is `FetchRequest`-shaped. The crawler needs its own concurrency/robots/frontier control, so `crates/apps/crawl/src/lib.rs:241` passes `ctx.engines.http.clone()` straight into `pumper_core::crawl(...)`, bypassing `AppContext::fetch` entirely. Cost recording (`costs.record`), tier learning (`tiers.record`), and budget governance live *inside* that one method rather than behind a reusable recording API, so any app that needs raw engine access silently loses all three. `AppContext.costs`'s own doc ("every metered engine call is attributed to this job", app.rs:46) and `fetch`'s "Prefer this over calling the fetcher directly" (app.rs:113) are both false for the platform's highest-volume app.
- **Impact**: The cost ledger and budget ceilings — a whole shipped feature area — are blind on the largest fetch path. Job `budget_usd` is unenforceable for `crawl` (a budgeted crawl can never abort, since nothing is ever recorded). `TierMemory` is starved of its richest signal: the crawl's per-host HTTP outcomes are exactly the data the adaptive router wants, and they are discarded.
- **Fix sketch**: Split the side-effects out of `fetch` into small public seams on `AppContext` and let raw-engine apps call them: (a) `pub async fn meter(&self, engine: &str, url: Option<&str>, cost_usd: f64, detail: Option<&str>)` wrapping the existing `self.costs.record(self.job_id, &self.app, ...)` block; (b) `pub async fn learn_tier(&self, host: &str, winner: &str, http_lost: bool)` wrapping the `self.tiers.record` block; then refactor `fetch` (app.rs:207-225) to call them so there is one implementation. Then have `crawl`'s call site wrap the `HttpClient` it passes at crates/apps/crawl/src/lib.rs:241, or accept an `Option<&AppContext>` sink in `CrawlConfig`, and meter per page. Metric: `SELECT app, engine, COUNT(*) FROM cost_events GROUP BY 1,2` should show non-zero `crawl` rows; `tier_memory` row count should grow after a crawl run.

## 2. Replace the per-call `SELECT SUM(cost_usd)` budget re-aggregate with an in-context running total

- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: hot-path
- **File**: crates/core/src/app.rs:84-90, 148-163, 207-225, 256-271
- **Scenario**: A budgeted job (`budget_usd = Some(_)`) makes N metered calls — e.g. `apps/research` or an `AutoWithResearch` fetch loop over a URL list. Every call re-reads the *entire* growing cost history for that job to compute headroom.
- **Root cause**: `remaining_budget_usd` (app.rs:84) delegates to `CostLedger::job_total`, which is `SELECT SUM(cost_usd) FROM cost_events WHERE job_id = ?1` (crates/core/src/costs.rs:84-91). It is called on the pre-flight path of both metered seams (`fetch` app.rs:149, `research`→`require_budget` app.rs:256), and each call re-scans all rows the job has written so far — classic O(n) per call → **O(n²) total** over a job. `idx_cost_events_job` bounds it to the job's own rows, so it's an index scan, not a table scan, but the growth is still quadratic. On top of that, `fetch` performs up to **four sequential SQLite round-trips per URL** — `tiers.preferred` (app.rs:130), `job_total`, `tiers.record` (app.rs:207), `costs.record` (app.rs:212) — three of which are pure telemetry that the caller awaits inline, contending for SQLite's single writer against concurrent jobs' dataset upserts (`worker.concurrency` defaults to 4).
- **Impact**: Bounded today only because the two `AppContext::fetch` callers (`apps/readable`, `apps/watch`) fetch once per job. It becomes the dominant overhead the moment finding #1 routes crawl volume through these seams — at 10k pages that is ~40k SQLite statements and a ~50M-row-visit quadratic SUM. Fixing it costs one atomic add per call.
- **Fix sketch**: Add `spent_usd: AtomicU64` (bit-cast f64, or `Mutex<f64>`) to `AppContext`, seeded once at context construction from `costs.job_total(job_id)` (covers retried jobs, whose prior attempts' events must still count). Have the two `self.costs.record(...)` sites (app.rs:212, 244, 261) `fetch_add` the recorded cost, and make `remaining_budget_usd` read the atomic instead of hitting the DB — keeping `job_total` for the API/reporting path. This preserves exact semantics (the ledger stays the source of truth on restart) while removing the per-call query. Metric: statements-per-fetch (target 4→1, or 4→0 on the pre-flight path) and wall-clock of a 500-URL budgeted `AutoWithResearch` loop.

## 3. Validate config invariants in `Config::normalize` — the heartbeat/reaper contract is documented but unenforced

- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/core/src/config.rs:56-82, 128-133
- **Scenario**: An operator tunes `[worker]` for long jobs and sets `heartbeat_secs = 300` but leaves `stale_after_secs` at its 120 default (or lowers it). Every running job's heartbeat is now *always* older than the stale threshold, so the reaper treats **every healthy in-flight job** as hung: re-queued mid-run, restarted, reaped again, until `max_attempts` is exhausted and it fails permanently. No job ever completes, and nothing in the logs points at the config.
- **Root cause**: `WorkerConfig::stale_after_secs`'s own doc-comment states the invariant — *"Must exceed `heartbeat_secs`"* (config.rs:132) — but it is a comment, not a check. `Config::load` (config.rs:58) only does TOML parsing plus `normalize()`, and `normalize()` (config.rs:77-81) does exactly one thing: the browser→http proxy fallback. There is no `validate()` anywhere in the crate (`validate_profile_name` in engine.rs is unrelated), so the config layer accepts any semantically-broken combination and the failure surfaces only as mysterious reap storms in `reap_once` (crates/server/src/worker.rs:305).
- **Impact**: A silent, total-availability misconfiguration with a one-line fix and no guard. The same gap covers sibling invariants that are equally unchecked: `job_timeout_secs` should exceed `stale_after_secs` (else the reaper races the timeout), `concurrency == 0` yields a worker that claims nothing, and `governor.penalty_base_secs > penalty_cap_secs` makes the cap meaningless. Cost of the fix: one function and a handful of unit tests, in a file that already has a test module (config.rs:471).
- **Fix sketch**: Add `fn validate(&self) -> Result<()>` to `impl Config`, called from `load()` right after `normalize()`, returning `Error::Config(...)` naming the offending keys and their values. Start with the three high-signal rules: `worker.heartbeat_secs > 0 && worker.stale_after_secs > 0 && worker.stale_after_secs <= worker.heartbeat_secs` → reject; `worker.job_timeout_secs <= worker.stale_after_secs` → reject; `worker.concurrency == 0` → reject. Follow the existing `Error::Config(format!("{}: {e}", path.display()))` message shape at config.rs:65 so the offending file is named. Add tests alongside `http_defaults_match_prior_hardcoded_values` asserting `Config::default().validate().is_ok()` plus one rejection case per rule.
