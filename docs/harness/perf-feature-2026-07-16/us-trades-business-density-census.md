## 1. Persist the saturation ranking and its ACS denominator instead of discarding them into job JSON

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/apps/census-density/src/lib.rs:301-350,603-672
- **Scenario**: The module doc states the point of this app is "the market-strength backbone for Ledgerline's geographic launch ranking … rank by SATURATION (establishments per 10k), not just absolute size." A consumer (the launch-ranking UI, a trigger, a webhook, an export) queries the datasets for saturation and finds nothing: `establishments` holds raw counts, `census/market_blend` holds `total_market` — neither carries a per-capita base. The only place saturation exists is the ephemeral `top_places_by_saturation` array in one job's result JSON, capped at 60 rows.
- **Root cause**: `fetch_denominator` pulls B01003_001E / B11001_001E / B25003_002E for every place and builds a `BTreeMap<String, Denom>`, then `run` uses it purely to compute an in-memory `saturation: Vec<Value>` for the return value. Nothing is upserted. The denominator is re-fetched from ACS on every run and thrown away; the derived `per_10k` never reaches a dataset, so change-detection, revisions, search, and triggers cannot see the app's headline metric. The blend (the durable cross-app artifact) was built at naics4×state grain but no denominator was ever joined to it.
- **Impact**: The stated primary output of the app is unqueryable and non-durable. Blocks the backlog's composite-launch-score, GeoJSON-export, and territory-white-space ideas, all of which need per-capita density as stored data. Also wastes one ACS round trip per run whose entire product is discarded. Fixing it also makes saturation available for the true (employer+solo) market, not just employer counts.
- **Fix sketch**: Add the denominator to the record payload rather than only the summary. In the CBP loop, carry `population`/`households`/`owner_occupied` and a derived `per_10k` onto each `all_records` entry (or into a sibling `census/saturation` dataset keyed `{place}` under `MARKET_APP`, alongside `MARKET_BLEND_DATASET`). Then extend `blend_market` to accept the `BTreeMap<String, Denom>` and emit `base`, `denominator_kind`, and `total_market_per_10k` on each `{naics4}:{state_fips}` cell — that is the number the launch ranking actually wants and it does not exist today. Keep the existing graceful degradation: a denominator-fetch failure should leave the base fields `null` with the reason on the record, not fail the upsert. Note `census-nonemp` also calls `sync_market_blend` without an ACS fetch, so the denominator join belongs inside the blend (reading the persisted base) rather than in the CBP loop only.

## 2. Stop full-scanning both datasets — the blend reads up to 100k records to use ~208 state rows

- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: batching
- **File**: crates/apps/census-density/src/lib.rs:406-428,487-490
- **Scenario**: A `geo=county` run over a few states writes ~4 NAICS × N counties into `establishments`. Every subsequent run of *either* Census app then calls `sync_market_blend`, which reads the entire `establishments` dataset (`BLEND_READ_LIMIT = 50_000`) plus the entire `nonemployers` dataset, deserializes every row's `data` into a `serde_json::Value` — and then `blend_market` immediately discards every county row at the `geo != Some("state")` guard.
- **Root cause**: `ctx.datasets.list(app, dataset, BLEND_READ_LIMIT)` is an unfiltered `SELECT … ORDER BY updated_at DESC LIMIT ?` — the geography predicate that decides what the blend can use lives in Rust, after the rows have already crossed the SQLite boundary and been parsed. `live()` then allocates a second `Vec<Value>` by moving each `r.data` out. The blend only ever needs state rows: at most 4 NAICS × 52 states = 208 employer rows and 2 × 52 = 104 solo rows. A nationwide county run (4 × 3,143 ≈ 12.5k rows) means ~98% of the deserialization work is thrown away, and the `ORDER BY updated_at DESC LIMIT 50000` means once the dataset exceeds the cap the blend silently reads an arbitrary recency window rather than the state rows it needs — a correctness cliff, not just a slow path.
- **Impact**: ~12.5k wasted `serde_json` row parses per blend on a nationwide county run, ×2 because both apps re-derive the blend. Bounded in absolute terms (these are annual jobs), but the fix is small, removes the 50k-cap correctness cliff, and cuts the read to the ~300 rows that matter.
- **Fix sketch**: `datasets.rs:656` already exposes `list_filtered(app, dataset, &[JsonFilter], after, limit)`. Read the employer side with `JsonFilter::Eq { path: "geo".into(), value: "state".into() }` so SQLite drops county rows before deserialization, and drop the now-dead `geo != state` guard in `blend_market` down to a defensive `debug_assert`. With the filter in place `BLEND_READ_LIMIT` can fall to a few thousand and stop being a silent truncation risk — or page with `list_filtered`'s `after` cursor for an exact read. Keep `blend_market` pure and `&[Value]`-shaped so the existing four unit tests stand unchanged.

## 3. Derive the employer-side wage and revenue benchmarks from the PAYANN/EMP already stored

- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/apps/census-density/src/lib.rs:230-231,257-271
- **Scenario**: A Ledgerline user asks "what does a plumbing shop in my state pay, and how big is a typical one?" `census-nonemp` answers the equivalent question for solo operators — it derives `avg_receipts_per_operator` and even a `national_avg_receipts_per_operator` for ranking. The employer side fetches `EMP` and `PAYANN`, stores both raw, and derives nothing.
- **Root cause**: Asymmetry between the two apps rather than a data gap. `census-density` parses `emp` and `pay` through `census_common::census_num`, writes `employees` and `annual_payroll_thousands` onto each record, and sums them into `total_employees` — but never computes the two ratios the inputs exist for. `trade_summaries` reports `total_establishments` / `total_employees` side by side without ever dividing them. The information is already fetched, already suppression-guarded, already persisted; only the derivation is missing.
- **Impact**: Zero new HTTP requests and no new API surface — two divisions unlock an average-wage benchmark and an average-shop-size metric per trade × place. This is the backlog's "Average-wage benchmark derived from CBP payroll" (idea 104, E2/I6/R2) and it is nearly free because the fetch already happened. It also gives the blend a natural employer-side complement to `solo_share`: solo receipts vs employer payroll per head.
- **Fix sketch**: In the CBP row loop where `emp` and `pay` are bound, add `avg_annual_wage` = `(pay * 1000) / emp` (PAYANN is in $1,000s, mirroring the `(rcpt * 1000) / estab` convention at `census-nonemp/src/lib.rs:186`) and `avg_establishment_size` = `emp / estab`, guarding `emp > 0` / `estab > 0` exactly as the solo side guards `estab > 0`. Important honesty constraint: `emp` and `pay` currently fall back to `0` via `.unwrap_or(0)` when their cell is suppressed, while `estab` is `continue`-skipped — so a suppressed-payroll row would compute a fabricated $0 wage. Emit `Value::Null` for the derived ratio when the input was suppressed, keeping the `unwrap_or(0)` only for the raw stored field and the totals. Roll a `national_avg_wage` into each `trade_summaries` entry to match the solo side's shape.
