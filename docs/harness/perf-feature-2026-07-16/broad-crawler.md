## 1. Stop re-serializing the entire frontier on every checkpoint — the save is quadratic over a crawl

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: hot-path / write-amplification / algorithmic-complexity
- **File**: crates/core/src/crawl.rs:635-641 (call site), 761-792 (`Checkpoint::save`), 699 (`CHECKPOINT_STRIDE`)
- **Scenario**: A resumable broad crawl (`checkpoint: "name"`, `max_pages` in the tens of thousands) across a large site. The crawl gets progressively slower as the frontier grows, and the checkpoint file is rewritten in full thousands of times.
- **Root cause**: `Checkpoint::save` builds a *complete* snapshot every time: `frontier.queue.iter().cloned()` + `frontier.seen.iter().cloned()` (up to `MAX_FRONTIER` = 100k `String`s) + `kept_hashes.to_vec()`, then `serde_json::to_vec` over the whole thing, then a full temp-file write + rename. It is fired on `stats.kept % CHECKPOINT_STRIDE == 0` — every 25 kept pages — so the per-save cost is O(|seen| + |queue| + |kept|), and total checkpoint work is O(max_pages/25 × frontier_size). It is also `.await`ed inline in the main loop, where `in_flight` is only polled at line 544, so every save stalls *all* concurrent fetches for its duration.
- **Impact**: A 100k-page crawl with a saturated 100k seen-set does ~4,000 saves × ~10 MB of JSON ≈ **40 GB of write amplification** and ~400M string clones, for state that changed by ~25 pages between saves. Each save also freezes the whole fetch pipeline. Metric to watch: bytes written to the checkpoint path per crawl, and p99 gap between successive `in_flight.next()` completions.
- **Fix sketch**: Two independent wins. (a) Decouple cadence from cost — replace the `kept % 25` trigger with a **time-based** gate (e.g. `tokio::time::Instant` + a ≥5s minimum interval), which alone cuts save count by orders of magnitude on a fast crawl. (b) Make the save cost proportional to the *delta*, not the frontier: have `Frontier` track newly-seen URLs since the last save and append them (plus new `dedup_index.hashes()` tail entries) to a JSON-lines journal, compacting to a full `Checkpoint` occasionally; `Checkpoint::load` replays journal + base. Bump `CHECKPOINT_VERSION` so the existing `CheckpointLoad::Incompatible` path handles old files. Cheaper interim step: move the existing full save off the loop via `tokio::spawn` with a single-flight guard, so in-flight fetches keep progressing during the write. Trade-off: a time-based stride widens the worst-case resume loss from 25 pages to a few seconds of crawling — acceptable, since the frontier's `seen` set already makes a resume idempotent.

## 2. Add a per-host page budget and host-fair frontier ordering

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap / crawl-coverage
- **File**: crates/core/src/crawl.rs:267-306 (`Frontier`), 103-137 (`CrawlConfig`), 504-534 (top-up loop)
- **Scenario**: A user seeds 20 competitor domains with `same_domain: false`… or even just `max_pages: 500` over 20 seeds, expecting broad coverage. Seed #1 is a large e-commerce site with 5,000 internal links on its homepage. The whole page budget is consumed by seed #1's depth-1 links; seeds #2-#20 are fetched once at depth 0 and their subtrees are never touched. The result reports a healthy `kept: 500` and `hosts: 20`, so the truncation is invisible.
- **Root cause**: `Frontier` is a plain FIFO — `push` appends to `queue.push_back`, `pop` takes `pop_front`. That is pure discovery-order BFS with **no notion of host at all**; `host_of` is only consulted downstream for robots/crawl-delay, never for ordering. `CrawlConfig` has `max_pages` (global) and `max_depth`, but no per-host cap. The gap exists because the frontier was built for the single-seed case, where FIFO-BFS is exactly right; the failure only appears once seeds fan out across hosts.
- **Impact**: Multi-seed and off-domain crawls are effectively single-site crawls. This blocks the comparison/monitoring use cases the product already leans on (the `pages` dataset + revisit sentinel are per-URL, so a starved host simply has no records to monitor). It also compounds the shipped crawl-delay work: a polite host with a 2.5s delay rotates to the *back* of a FIFO dominated by one fast host, so it is starved further.
- **Fix sketch**: Add `max_pages_per_host: Option<usize>` to `CrawlConfig` and a `per_host: HashMap<String, VecDeque<(String, u32)>>` + round-robin cursor inside `Frontier`, keeping the existing `seen` set and `dropped` counter unchanged. `push` routes by `host_of(&url)`; `pop` rotates to the next host with a non-empty queue (which also naturally skips hosts sitting inside their `next_allowed` window, replacing part of the `rotations` scan at line 504). Track a per-host kept count and add `skipped_host_budget` to `CrawlStats` so truncation is reported honestly, matching the existing `frontier_dropped` precedent. Note this is *not* the backlogged "best-first crawl scoring" idea (#211) — that ranks URLs by relevance; this is host fairness and is orthogonal (and a prerequisite for it to be useful across hosts). The `Checkpoint` shape changes, so bump `CHECKPOINT_VERSION`.

## 3. Use sitemap `<lastmod>` to skip and prioritize — it is parsed away today

- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: feature-gap / automation
- **File**: crates/core/src/crawl.rs:1135-1140 (`parse_sitemap_locs`), 1145-1197 (`seed_from_sitemaps`), 76-81 (`RevisitSeed`), 55 (`CrawlPageRecord::simhash`)
- **Scenario**: A user runs the shipped revisit sentinel (`mode: "revisit"`) nightly over a 10k-page news/docs site that publishes a well-maintained sitemap. Every sweep issues 10k conditional GETs, even though the sitemap says only 40 URLs changed since yesterday. The `max_pages` cap then spreads the sweep across multiple runs, delaying detection of the changes that *do* matter.
- **Root cause**: `parse_sitemap_locs` is a single regex over `<loc>` only — `<lastmod>`, `<changefreq>` and `<priority>` are discarded before `seed_from_sitemaps` ever sees them. Sitemap seeding was built purely as a *seed-expansion* feature (turn a sitemap into a URL list), so the freshness metadata sitting right next to each `<loc>` was never plumbed through. Nothing downstream can use it: `RevisitSeed` carries only `url` / `etag` / `last_modified`, and `CrawlPageRecord` has no crawl timestamp.
- **Impact**: The sentinel does ~250× more network work than the site itself says is necessary on a typical low-churn day (10k conditional GETs vs ~40). Even a `304` costs a full connection + round-trip and is subject to per-host crawl-delay, so this is the dominant cost of a sentinel sweep, and it is what forces the 10k-seeds-per-run cap noted in `docs/features/crawling.md` § Known gaps.
- **Fix sketch**: Change `parse_sitemap_locs` to return `Vec<SitemapEntry { loc, lastmod: Option<String> }>` — parse `<url>`…`</url>` blocks and pull the optional `<lastmod>` (W3C datetime) from each. Two consumers: (a) **prioritize** — in `seed_from_sitemaps`, push newest-`lastmod` entries first so a `max_pages`-capped crawl spends its budget on fresh URLs; (b) **skip** — add a `last_crawled` timestamp to `CrawlPageRecord`/`RevisitSeed` and, in revisit mode, skip the conditional GET entirely when `lastmod <= last_crawled`, counting it in a new `skipped_lastmod` stat. Keep the existing `MAX_SITEMAP_SEEDS` / `MAX_SITEMAPS_PER_HOST` caps and the `filter.allows` gate as-is. Trade-off: sitemap `lastmod` is self-reported and some sites stamp it to "now" on every build — so gate the *skip* behavior behind an opt-in flag (`trust_lastmod`, default false) while making the *prioritization* unconditional, since mis-ordering is harmless.
