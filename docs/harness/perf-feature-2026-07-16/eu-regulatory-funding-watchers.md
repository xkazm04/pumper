## 1. Wire eu-sedia into `grants/unified` with a real `normalize_eu_sedia`

- **Severity**: Critical
- **Lens**: feature-scout
- **Category**: coverage-gap
- **File**: crates/apps/eu-sedia/src/lib.rs:138-152 (and crates/apps/eu-sedia/Cargo.toml:6-9)
- **Scenario**: A user hits `GET /grants?status=open&close_date_before=…` or `GET /grants/closing-soon?days=14` expecting the pan-EU corpus the app advertises ("one source that serves every EU member state"). They get grants.gov + ca-grants only. Every Horizon Europe / Erasmus+ / CERV / LIFE topic pumper fetches daily is invisible to the product's entire query surface, to cross-source dedup, and to per-opportunity full-text search.
- **Root cause**: Still genuinely open — verified against code, not inherited from the prior report. `crates/apps/eu-sedia/Cargo.toml` has no `grants-common` dependency (ca-grants:9 and grants-gov both do). `eu-sedia::run` ends at `ctx.upsert_many("opportunities", &records)` (lib.rs:138) and returns a hand-built result object; it never calls `grants_common::finalize_unified`, and `grants-common/src/lib.rs` exposes only `normalize_grants_gov` (:74) and `normalize_ca_grants` (:110) — there is no `normalize_eu_sedia`. `docs/features/apps.md:22-23` confirms the split: unified is documented as grants-gov + ca-grants, and `routes.rs:2643` documents the `source` filter as exactly `grants-gov | ca-grants`. eu-sedia predates the unified layer and was never backfilled.
- **Impact**: The largest and most differentiated source in the fleet (27 member states, up to 1000 topics/run under current caps) contributes **zero rows** to `grants/unified`. Concretely it is excluded from: (a) `GET /grants` SQL filtering; (b) `GET /grants/closing-soon` — the *cross-source* deadline view; (c) `sweep_closed` lifecycle, so no EU topic ever transitions to `closed`; (d) `link_duplicates` at `DUP_DISTANCE` — and EU↔national-portal syndication is precisely the near-duplicate case SimHash exists for; (e) `index_datasets` (`UnifiedOutcome::merge_into`, grants-common:45-48), so EU topics are **not individual full-text search docs** and cannot be alerted on by saved searches; (f) `drift_warnings`. Plumbing cost is now one dep + ~40 lines, because `finalize_unified` owns all four calls.
- **Fix sketch**: Add `grants-common.workspace = true` to eu-sedia's Cargo.toml; add `pub fn normalize_eu_sedia(hit: &Value) -> Option<(String, Value)>` to `grants-common/src/lib.rs` beside the other two; in `run`, build `unified_items` from the same `hits` and call `finalize_unified(&ctx, &unified_items).await?`, then `cross.merge_into(&mut out)` (mirror grants-gov:176-208). Field mapping from the real SEDIA `metadata` shape read by `normalize` (lib.rs:172-205) — note SEDIA metadata values are arrays, so reuse the local `first()` accessor:
  - `source` → `"eu-sedia"`; `source_id` → `first(m,"identifier")` (fallback `hit.reference`, same key as lib.rs:175); key = `format!("eu-sedia:{id}")`
  - `title` → `first(m,"title")` **via `clean_inline`** — raw SEDIA titles are entity-escaped (`&amp;`, `&#8211;`), and unified must not carry tag soup into the search index
  - `agency` → `first(m,"frameworkProgramme")` (e.g. "Horizon Europe"); consider appending `first(m,"callIdentifier")` — SEDIA has no agency column, the programme is the honest analogue
  - `status` → **needs an explicit code map before `norm_status`**: SEDIA statuses are numeric codes (`31094502`=open, `31094501`=forthcoming, per lib.rs:39-41), and `norm_status` (grants-common:417) passes unknowns through lowercased — mapping naively would write the literal `"31094502"` into `status` and silently break every `?status=open` filter and `sweep_closed`'s `matches!(status, Some("open")|Some("forecasted"))`. Map `31094502`→`open`, `31094501`→`forecasted`.
  - `open_date` → `first(m,"startDate")` through `norm_date`
  - `close_date` → `deadlineDate` is an **array kept whole** (lib.rs:200) because Horizon topics have multi-stage/multi-cutoff deadlines. Take the earliest element that parses and is ≥ today (fall back to the last element) — do **not** take `[0]` blindly, or a passed first cutoff will make `sweep_closed` flip a still-open two-stage call to `closed`. `parse_date` already handles the ISO/`T`-datetime forms.
  - `award_floor`/`award_ceiling`/`total_funding` → **leave `Value::Null` initially.** `budgetOverview` is EUR and the unified schema has no currency dimension, so running it through `money_range`/`money_scalar` would file €10M as if it were the same unit as ca-grants' dollars and corrupt `GET /grants?min_award=`. Null is the honest mapping until unified gains a `currency` field; call this out in the doc rather than parsing.
  - `categories` → `str_list`-style array from `first(m,"typesOfAction")` + `first(m,"programmePeriod")`; `eligibilities` → `[]` (not in search hits); `aln` → `Null` (US-only concept)
  - `url` → `hit.url`; `description` → the already-computed `description_text` truncated to 500 chars (matches `normalize_ca_grants`:134-135)
  - Trade-off to state in `docs/features/apps.md`: as with the CA field additions, the first run after deploy writes ~1000 new unified rows and re-links duplicates; `sweep_closed`'s full-table read now runs on a third schedule (see finding 2's note).

## 2. Stop paging a match-all query with no sort — the 1000-row window is non-deterministic, not just truncated

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: incremental-fetch
- **File**: crates/apps/eu-sedia/src/lib.rs:86-136 (caps at :56-67, break at :132-135)
- **Scenario**: A daily 10:00 UTC run POSTs `text=***` (match-all, required by the API) with `pageSize=100`, `maxPages=10` and no sort key, then stops at `pages_fetched >= max_pages`. When SEDIA's `totalResults` exceeds 1000 — routine for Horizon Europe alone across `type` 1+2 — the run keeps whichever 1000 topics the server happened to rank first *that request*.
- **Root cause**: Two compounding gaps. (a) The URL (`:87-89`) passes only `apiKey`/`text`/`pageSize`/`pageNumber` — no `sortBy` — so page order is the server's relevance `weight`, which this module's own header calls volatile (`:15`). Pagination over an unsorted, server-scored match-all has no stable window. (b) The break at `:132` treats `pages_fetched >= max_pages` identically to genuine exhaustion: no error, no warning, no `truncated` flag. `totalResults` and `fetched` both land in the result (`:144-146`), so the evidence is present but nothing acts on it. There is also no delta narrowing — the ES `bool.must` (`:70-75`) filters only `type` and `status`, so every run re-fetches, re-normalizes, and re-hashes the entire corpus to discover a handful of changes.
- **Impact**: Change-detection thrash, which is worse than plain truncation: topics drift in and out of the captured window between runs, so `upsert_many` reports phantom `new` rows for topics that were merely outside yesterday's window, while topics that fall out are silently frozen at stale content forever (upsert-only — absence is never a delisting, per grants-common:148-155). Every downstream signal built on `new`/`changed` — and, once finding 1 lands, `sweep_closed` and the duplicate links — inherits the noise. Cost side: ~10 POSTs/run each returning 100 hits whose `descriptionByte` is full topic HTML (Expected Outcome/Scope, tens of KB each) ≈ tens of MB fetched and re-hashed daily for a corpus that changes by a few rows.
- **Fix sketch**: Three steps, cheapest first. (1) **Report the truncation**: capture `let truncated = pages_fetched >= max_pages && (pages_fetched * page_size) < total;` at the break and add it plus a `warnings` entry to the result — a silently-partial corpus must not read as a clean success (same principle as the existing drift guard at `:113-121`). (2) **Stabilize the window**: add a deterministic sort to the query string (`sortBy`/`sortOrder` on `identifier`) so page N means the same rows across requests; verify the parameter name against the saved `page1.json` artifact / a live probe before shipping — do not guess the upstream contract. If SEDIA exposes no stable sort, raise the default `maxPages` so the cap is not reached in practice (the clamp already allows 50 → 5000 topics) and keep the truncation flag as the tripwire. (3) Cheap allocation win while in here: `:107-111` deep-clones the whole `results` array (every hit's `descriptionByte` HTML) purely to iterate it at `:122` — take `parsed.get("results").and_then(Value::as_array)` by reference and clone only the fields `normalize` keeps. Secondary; do it as part of (1)/(2), not on its own.

## 3. Make the CMS release watcher self-baselining instead of trusting a hand-edited `known_release`

- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: automation
- **File**: crates/apps/cms-fee-schedule/src/lib.rs:141-143, 187-213
- **Scenario**: A scheduled run (monthly, 06:00 on the 1st) uses `default_params` — `known_release: "RVU26A"`, a string literal. CMS ships RVU26B; the app correctly reports `is_newer_than_known: true`; someone re-runs `scripts/ingest-cms-pfs.mjs` and Counterbill is now current. The literal still says `RVU26A`, so every run from then on reports `is_newer_than_known: true` forever. The one field named after the user's actual question — "am I stale?" — becomes a permanently-lit alarm that gets ignored, and the *next* real release lands in a channel nobody trusts.
- **Root cause**: The baseline is a param, not state. `known_release` is only read (`:188`) and compared (`:189-192`); nothing ever writes it back, and no scheduled run can supply it since scheduled runs take `default_params`. `None` degrades to `is_newer = true` (`:191`) — a sensible bootstrap, but it makes the stale-literal and no-baseline cases indistinguishable. The app also already holds the better signal it isn't leaning on: `ctx.upsert("releases", schedule, &record)` at `:185` returns a `ChangeKind`, and `change_since_last_run`/`is_fresh` fire exactly once, on the run where CMS actually publishes. Separately `:151-155` hard-errors on any `schedule != "pfs"` while the header (`:17-19`) and `zip_url`/`source_url` conventions are already parameterized — the clfs/asp extension is scaffolded but unbuilt.
- **Impact**: Bounded but corrosive: one false-positive freshness field on a quarterly-cadence watcher whose entire purpose is being believed. No data loss — `is_fresh` remains correct. Fixing it also unlocks the real automation: this app is a textbook `dataset` trigger source (`docs/features/triggers.md`), yet its output is `ingest_hint` (`:208-213`) — a prose sentence telling a human to go edit a script.
- **Fix sketch**: (a) Read the baseline from storage rather than params: before the upsert, fetch the existing `releases`/`pfs` row and use its `latest_release` as the implicit baseline, keeping the `known_release` param as an explicit override for callers who know what Counterbill has baked. Drop the stale `"RVU26A"` literal from `default_params`. (b) Distinguish the states honestly: report `baseline_source: "param" | "stored" | "none"` alongside `is_newer_than_known` so a consumer can tell a real gap from a cold start. (c) Add a structured `ingest` object beside the prose hint (`{release, zip_url, source_url}`) so a `dataset` trigger on `app=cms-fee-schedule, dataset=releases, on_change=fresh` can fan out to an ingest target that reads the keys from `_trigger` — turning "a human notices the hint" into a pipeline edge. (d) The clfs/asp generalization is a separate, larger unit — leave `:151-155` as an honest hard error until a second schedule's URL conventions are verified, rather than half-parameterizing it now.
