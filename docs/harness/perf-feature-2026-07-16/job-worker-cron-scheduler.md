## 1. Index only the records this run changed instead of re-indexing the whole dataset

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: n-plus-one / write-contention
- **File**: crates/server/src/worker.rs:530-555 (call site :231-235)
- **Scenario**: `grants-gov` (daily 09:00) and `ca-grants` (daily 09:30) both name `grants/unified` in their result's `index_datasets` (docs/features/apps.md:22). A normal incremental sync changes a handful of opportunities — but every run re-reads and re-writes the *entire* corpus into Tantivy. `docs/harness/harness-learnings.md:52` already records this as a known limit; it has not been fixed.
- **Root cause**: `dataset_search_docs` unconditionally calls `state.datasets.list(app, dataset, 100_000)` — a full `SELECT … FROM records WHERE app=?1 AND dataset=?2 ORDER BY updated_at DESC LIMIT 100000` (crates/core/src/datasets.rs:605) — then builds a `SearchDoc` per live record. `TantivyIndex::index` (crates/engine-search/src/lib.rs:142-164) then does `delete_term` + `add_document` for every doc and commits. Nothing consults which records actually changed, even though the run's revisions are loaded ~15 lines later by `load_run_changes` (worker.rs:334-346) for watches and triggers.
- **Impact**: Per run, per named dataset: ~5.2K rows loaded + JSON-parsed into `Record`, ~5.2K `rec.to_string()` re-serializations, ~5.2K delete+add ops and a full segment-merging commit — for typically single-digit changed rows (the `reindex` bin rewrote 5196 rows, per prior campaign notes). That is roughly a 100–1000× write amplification on the search index, ×2 apps ×daily, all on the job's critical path before `storage.complete`. It also grows linearly with the corpus, so the cost of every run rises forever.
- **Fix sketch**: Hoist `load_run_changes(&state, &job)` above the indexing block and pass its revisions into `dataset_search_docs`. For each `index_datasets` spec, intersect the run's revisions with that `app`/`dataset` (reuse `group_by_dataset`), then `datasets.get(app, dataset, &rev.key)` (crates/core/src/datasets.rs:~590) only the touched keys to build docs; route revisions whose record is `removed_at`-set to `Search::delete_ids` (engine-search/src/lib.rs:166) instead of leaving stale docs indexed — which the current `continue`-on-removed path silently does. Keep the full-corpus path available behind an explicit `"reindex": true` spec flag and the existing `--bin reindex` for backfills. Stays fail-open per the worker convention. Metric to watch: docs-indexed-per-job and `state.search.index` wall time, both of which should drop to O(changes).

## 2. Stop re-enumerating the whole missed-firing backlog on every scheduler tick

- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: polling
- **File**: crates/server/src/scheduler.rs:167-191 (callers :70-102)
- **Scenario**: A frequent schedule (say `*/60 * * * * *`) whose app run is long — or simply one whose previous run is still active — hits the overlap guard at scheduler.rs:99-102, which deliberately does **not** touch `last_run` so the firing stays due. Every subsequent tick therefore re-evaluates `decide` against the same stale `reference`, and the gap it must enumerate grows for as long as the run lasts.
- **Root cause**: `decide` walks `cron.after(&reference_tz)` counting *every* firing in `(reference, now]` up to `MAX_MISFIRE_SCAN` (10_000) — but `count` is only load-bearing for `Action::Skip { missed }`. The `Fire` arm uses it purely as a log field (`collapsed`, scheduler.rs:116). The loop cannot early-exit because it computes an exact count nobody needs. Separately, `CronSchedule::from_str(&schedule.cron)` (scheduler.rs:70) re-parses each schedule's expression from scratch on every tick, and `reconcile` walks all schedules serially.
- **Impact**: A per-minute schedule blocked by a 6-hour run enumerates ~360 cron firings per tick, rising to the 10_000 cap; at the default tick that is 10⁴ chrono/DST-aware iterations per schedule per tick, plus one `CronSchedule::from_str` parse per schedule per tick (200 schedules = 200 parses/tick). Because `reconcile` is serial, this inflates tick latency and delays *unrelated* due schedules — the tick's own cost is what makes on-time firings late.
- **Fix sketch**: In `decide`, take `earliest` via `cron.after(&reference_tz).next()`, then compute `missed` from `earliest` and `grace` as today. Only enumerate further when `missed && misfire_skip` (the sole consumer of an exact count); for the `Fire` path bound enumeration to a tiny cap (e.g. 2) and report `collapsed` as a saturating "n+" for logs. This drops the common path from O(missed firings) to O(1) with no behavior change — the existing unit tests (`fire_once_collapses_a_downtime_backlog_into_one_run`, `skip_advances_past_a_downtime_backlog_without_running`) pin the semantics, though the former's exact `collapsed: 3` assertion would need to reflect the new bound. Additionally, cache parsed `CronSchedule` by expression string across ticks (a small `HashMap<String, CronSchedule>` owned by `run`), keyed so an edited cron re-parses. Trade-off: `collapsed` becomes approximate in logs — acceptable, it is diagnostic only.

## 3. Make schedules observable: computed `next_run`, last outcome, and a "why am I not firing" reason

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/server/src/scheduler.rs:70-102 (model: crates/core/src/storage.rs:63-80; API: crates/server/src/routes.rs:953-969)
- **Scenario**: A user runs unattended periodic scrapes — the product's core promise. They want to answer three routine questions from the API: "when does this schedule next fire?", "did its last run succeed?", and "why has this one gone quiet?". Today none of them are answerable. `GET /schedules` returns only `id, app, cron, params, enabled, priority, timezone, misfire_policy, max_attempts, last_run, created_at`. A schedule can be silently dead for days and look perfectly healthy.
- **Root cause**: Every reason a schedule fails to fire is written to a **server log line and nowhere else**: invalid cron (`warn!` scheduler.rs:73), app not in the registry (`warn!` scheduler.rs:93), and previous run still active (`info!` scheduler.rs:100). The overlap guard is the sharpest case — by design it does not touch `last_run`, so a schedule wedged behind a stuck run shows an ever-older `last_run` with no indication that the scheduler is deliberately holding it back. The next firing is never persisted or returned even though `decide`/`CronSchedule` computes it every tick, and the `schedules` table has no link back to the jobs it enqueued — despite `EnqueueOptions.schedule_id` (scheduler.rs:110) already stamping the association, which `schedule_has_active_job` (crates/core/src/storage.rs:495) proves is queryable.
- **Impact**: Operating N schedules requires grepping server logs; the vision-scan backlog has 7 scheduler/worker ideas (T8) but none covers this, and `/metrics` (routes.rs:294-299) only exposes enabled/disabled *counts*. Silent non-firing is the single failure mode that most directly negates the product's value, and it is currently invisible over the API.
- **Fix sketch**: Three increments, each independently shippable. (a) **`next_run`, zero schema change**: in `list_schedules` (routes.rs:953), parse each row's cron and reuse the scheduler's own reference rule (`last_run.unwrap_or(created_at)`) to project `cron.after(reference).next()` in the schedule's `parse_tz(timezone)` — return it as a computed field. This is pure and shares `decide`'s exact semantics, so the API can never disagree with the scheduler. (b) **Last outcome**: join the newest job carrying this `schedule_id` (the index behind `schedule_has_active_job` already supports it) and surface `last_job_id` + `last_status`. (c) **Health reason**: return a `health` enum — `ok` / `invalid_cron` / `unregistered_app` / `overlapping` — derived from the same three checks at scheduler.rs:70-102, so the log lines and the API are computed from one source. Natural follow-on once (b) lands: auto-disable (or fire the global `[webhooks] failure_url`) after K consecutive failed runs, which turns a silently-broken scraper into a push notification rather than something a user discovers late.
