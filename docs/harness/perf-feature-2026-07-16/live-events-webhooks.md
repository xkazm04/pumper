## 1. Share `JobEvent` behind an `Arc` and byte-budget the replay ring

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: fan-out / unbounded-buffer
- **File**: crates/server/src/events.rs:44,89-103
- **Scenario**: An extractor or crawl job finishes with a multi-MB `result` JSON. `finalize` (worker.rs:470-473) copies `job.result` into the `JobEvent` and calls `EventBus::emit`. Meanwhile 20 dashboards hold `/events` and 50 clients hold `/jobs/{id}/stream`; every in-flight job also emits a `progress` event with a full snapshot every 2s (progress.rs:107-110).
- **Root cause**: `SeqEvent = (u64, JobEvent)` is a by-value type, and `JobEvent.result` is an owned `serde_json::Value`. `emit` does `ring.push_back((seq, event.clone()))` — a deep recursive clone of the whole `Value` tree — then `tx.send((seq, event))` moves a second copy into the broadcast slot, and tokio's broadcast clones the value **again per receiver** at `recv()`. The ring (`EVENT_RING_CAPACITY = 1024`, state.rs:24) and the broadcast buffer (512, state.rs:21) are bounded by **event count, not bytes**, so they pin up to 1024 + 512 full result payloads in RSS for the process lifetime, with no eviction pressure from memory. Worse, `stream_job` subscribes to the *global* bus and filters with `event.job_id != id` (routes.rs:397) — so each of the 50 per-job streams deep-clones every unrelated job's result only to discard it.
- **Impact**: With 1MB results, the ring alone is ~1GB of resident memory that never shrinks; the deep clone is O(size of result) × (1 ring copy + 1 send copy + N subscriber copies). At 70 subscribers that is ~72 copies per event where 1 shared allocation suffices — the dominant CPU and allocator cost on the event hot path. Metric to watch: RSS after a burst of large-result jobs, and `emit` p99 latency (it runs under the ring `Mutex`, so the clone serializes all emitters).
- **Fix sketch**: Change `pub type SeqEvent = (u64, Arc<JobEvent>);`. In `emit`, build `let event = Arc::new(event);` once, push `(seq, Arc::clone(&event))` to the ring and `tx.send((seq, event))` — ring, broadcast slot, and every subscriber then share one allocation and per-receiver `recv()` becomes an `Arc` refcount bump. `Replay::Events(Vec<SeqEvent>)`, `resume`/`recover`/`sse_event` in routes.rs take `&JobEvent` already, so they need only deref changes. Separately, add a byte budget to `EventBus`: track a running `bytes: usize` (approximate each event by `result` serialized length, computed once at emit) and evict from the front while `bytes > MAX_RING_BYTES` (e.g. 32MB) in addition to the `capacity` check — the existing `Replay::Reset` path already handles the resulting older evictions correctly. Optionally cap `result` size in the event itself (emit a truncation marker; `GET /jobs/{id}` already serves the full result).

## 2. Auto-drain the webhook dead-letter queue instead of requiring a manual replay

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap / automation
- **File**: crates/server/src/webhook.rs:19,169-201
- **Scenario**: A consumer's webhook receiver restarts for 30 seconds during a deploy. Every job that finishes in that window burns its 3 attempts (`MAX_ATTEMPTS = 3` with linear backoff = 0s, 2s, 4s — a ~6s total window) and lands in `webhook_deliveries` as `failed`. The consumer, whose whole reason for using webhooks is "never have to poll" (module doc, webhook.rs:1-7), silently misses those results forever unless a human notices the DLQ and POSTs `/webhooks/deliveries/{id}/replay` for each row.
- **Root cause**: The DLQ and the replay path exist (`webhook::replay`, routes.rs:1961-1990) but replay is **only** operator-triggered. Nothing scans for `status = 'failed'` rows: `scheduler.rs` has no delivery handling, and `deliver`'s retry budget is a fixed ~6s in-process loop that dies with the spawned task. A receiver outage longer than 6 seconds is indistinguishable from a permanently bad URL.
- **Impact**: Any receiver downtime > ~6s causes permanent, silent event loss for every job finishing in that window — the failure mode the webhook contract exists to prevent. For a fleet completing 100 jobs/min, a 5-minute receiver restart drops ~500 deliveries that all require manual replay.
- **Fix sketch**: Add a `next_retry_at` + `retry_count` column to `webhook_deliveries` (new migration) and a periodic drain task alongside the scheduler: select `status = 'failed' AND retry_count < N AND next_retry_at <= now()` (small batch, ordered by `next_retry_at`), resolve the secret exactly as `replay_delivery` does today, and hand each to `webhook::replay`. Have `log_outcome` set `next_retry_at = now + backoff(retry_count)` with exponential backoff and jitter (e.g. 30s → 1m → 5m → 30m → 2h, capped at ~6 attempts), and mark rows `dead` past the cap so the DLQ view stays meaningful. Factor the secret-resolution `match delivery.kind` out of `replay_delivery` into a shared `resolve_secret(&storage, &delivery)` so the route and the drain task cannot drift. Pairs naturally with the documented retention/purge gap (events-webhooks.md:29) — the same task can purge `delivered` rows past a TTL.

## 3. Send a delivery id and timestamp, and sign them with the body

- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: feature-gap / integration contract
- **File**: crates/server/src/webhook.rs:171-201
- **Scenario**: A consumer receives `job.terminal` for job X, times out replying 200 after acting on it, and pumper retries — so the consumer processes X twice. Later an operator hits `/webhooks/deliveries/{id}/replay` and the identical body arrives a third time. The receiver has no way to recognize any of these as the same delivery, because every attempt is byte-identical with an identical signature.
- **Root cause**: `deliver` sends only `content-type`, `x-pumper-event`, and `x-pumper-signature`. The signature is `HMAC(secret, body)` (webhook.rs:178, 203-207) with no timestamp and no per-delivery nonce, and the `delivery_id` that `spawn_logged`/`replay` already hold is never put on the wire. So the receiver gets no idempotency key and no replay window: a signed body captured off the wire stays valid forever and re-POSTs as authentic.
- **Impact**: Bounded but real — at-least-once delivery with no dedup key pushes idempotency onto every consumer (and `dispatch_change`/`search.matched` payloads have no natural unique key at all, unlike `job_id`). Also means a captured signed request can be replayed indefinitely against the receiver.
- **Fix sketch**: Thread `delivery_id` into `deliver` and add `x-pumper-delivery-id: <id>` plus `x-pumper-timestamp: <unix secs>` headers. Change `sign` to a versioned signature base — `sha256=hex(HMAC(secret, format!("{ts}.{delivery_id}.").as_bytes() ++ body))` — so the timestamp and id are covered and the receiver can reject stale timestamps. Compute the signature per attempt (it currently hoists out of the retry loop at webhook.rs:178, which is correct for a body-only scheme but must move inside once the timestamp is signed) and keep the `delivery_id` **stable across retries and replays** — that stability is exactly what makes it a usable idempotency key. `replay` already receives `delivery_id` (webhook.rs:105-118); `spawn_logged` has it after `create_delivery`, and the fallback path at webhook.rs:139-141 (log write failed) can send a generated uuid. Document the header contract and the signature base in `docs/features/events-webhooks.md` § "Outbound webhooks".
