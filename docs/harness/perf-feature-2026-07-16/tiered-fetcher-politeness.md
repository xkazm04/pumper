## 1. Extend the politeness governor to the browser tier (today it polices only HTTP)

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/core/src/governor.rs:1-5, 89-126; escalation at crates/core/src/fetcher.rs:331-408; sole call site at crates/engine-http/src/lib.rs:390-414
- **Scenario**: `Governor::acquire` / `penalize` / `reward` are called from exactly one place in the workspace — `HttpEngine::send` (`grep -rn "governor" crates/engine-browser/` returns **zero hits**). So every browser render goes out with no per-host spacing, no jitter, and no learned penalty. Two paths make this the *common* case, not an edge: (a) an `Auto` escalation where the HTTP tier was thin/blocked immediately re-hits the same origin via Chrome with zero spacing; (b) the shipped tier router pins a host to `skip_http` after 3 HTTP losses (fetching.md:99-103), after which **all** traffic to that host is browser-tier and therefore entirely unpoliced. The hosts that get pinned are precisely the ones that already blocked us.
- **Root cause**: the governor was built as an HTTP-engine concern and wired inside `HttpEngine`, rather than at the tier seam that owns "one origin, N engines". `governor.rs:1-5` claims "a tiered escalation (http -> browser -> claude) ... never hammers a single origin" and fetching.md:95-97 describes it as a per-host token bucket — both are false for the browser tier. The governor also cannot learn from browser-tier blocks: `challenge_marker` fires in `fetcher.rs:348` and nothing calls `penalize`, so a Cloudflare wall seen by Chrome teaches the platform nothing.
- **Impact**: politeness/ban risk on exactly the hosts already hostile to us. A `skip_http`-pinned host with `default_rps = 5` receives *unlimited* browser renders — the configured rate limit silently stops applying. Also a learning hole: browser-tier `Blocked` verdicts (already detected and traced) never feed `penalize`, so the adaptive backoff that shipped is blind on the tier where blocks concentrate.
- **Fix sketch**: move the `acquire` call up to `Fetcher::fetch` (parse host once from `req.url`, `governor.acquire(host).await` before each tier attempt) and drop it from `HttpEngine::send` — or, less invasively, thread `Arc<Governor>` into `BrowserEngine::render` alongside the existing holder `acquire`. Then close the learning loop: where `fetcher.rs:374-386` records a browser `TierVerdict::Blocked`, call `governor.penalize(host, None)`; where it records `TierVerdict::Ok`, call `governor.reward(host)`. `snapshot_penalties`/`restore_penalty` then persist browser-learned penalties for free. Note the trade-off: spacing browser renders makes escalations strictly slower by one interval — that is the intended cost of politeness, and `Fetcher`-level placement avoids double-charging the HTTP tier.

## 2. Revalidate expired cache entries with the stored ETag instead of re-downloading the full body

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: caching
- **File**: crates/core/src/cache.rs:74-99 (get), 102-127 (put); miss path at crates/engine-http/src/lib.rs:537-568
- **Scenario**: `HttpCache::get` selects `WHERE key = ?1 AND expires_at > ?2` — an expired row is a plain miss, and `HttpEngine::fetch` then runs a full unconditional `send()`. The row's `headers` JSON already contains the origin's `ETag` / `Last-Modified` (whole response headers are stored at cache.rs:108), and the engine already knows how to do conditional GET (`HttpRequest.etag` → `if-none-match`, engine-http:348-352, with the 304 pass-through at 545-548) — but nothing joins the two. The validators are written to disk and never read back. The `watch` app (poll the same URL forever, `to_markdown = true`) is the canonical victim: every poll past the TTL re-downloads the entire body, even for a page that hasn't changed since the last poll.
- **Root cause**: conditional GET was built for the crawler's revisit mode, where the *caller* owns the validator and wants the raw `304` (which is why `fetch` returns the 304 straight through rather than resolving it against the cache). The cache was built as a pure TTL store. Neither layer was given the "expired-but-maybe-still-valid" state, so freshness is binary and revalidation is caller-only.
- **Impact**: on a monitor/poll workload the unchanged-page ratio is high (that's why change-detection exists at all), and each false miss costs a full body transfer — up to the 16 MiB `max_body_bytes` cap, low-single-digit MiB for real targets like SEDIA/census blobs — where a `304` is a few hundred bytes. It also cascades into finding #3: a revalidated hit skips the `html_to_markdown` parse entirely. Metric to watch: bytes-in per fetch and the ratio of `304`-resolved to full-body responses on `watch`-driven hosts; expect the p50 poll of an unchanged page to drop from body-transfer + parse to one round trip.
- **Fix sketch**: add `HttpCache::get_stale(key) -> Option<(HttpResponse, validators)>` (same SELECT minus the `expires_at` predicate, pulling `etag`/`last-modified` out of the already-deserialized `headers` map), plus `HttpCache::refresh(key, ttl)` (an `UPDATE http_cache SET expires_at = ?, created_at = ?` — no body rewrite). In `HttpEngine::fetch`, when `get` misses and `req.etag`/`if_modified_since` are unset, consult `get_stale`, populate the validators on the outgoing request, and on a `304` call `refresh` and return the stored body with `cache_hit: true`. Keep the existing raw-304 pass-through for callers that set validators themselves, so the crawler's revisit contract is untouched. `created_at` and the `max_age` staleness cap (cache.rs:79-81) both keep working, since `refresh` moves `created_at` forward only on a confirmed-still-valid revalidation.

## 3. Stop building a full Markdown document just to count characters on `Auto` fetches

- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: hot-path
- **File**: crates/core/src/fetcher.rs:263-269 (HTTP tier), 349-352 (browser tier), discarded at 502-503
- **Scenario**: on the escalating strategies, `needs_count` is true, so `(req.to_markdown || needs_count).then(|| html_to_markdown(&resp.body))` runs unconditionally — a full `scraper::Html::parse_document` + tree walk + `normalize` (markdown.rs:17-23) that allocates a Markdown `String` roughly the size of the page's text. Its only consumer is `text_len.map_or(true, |n| n >= min_chars)` against a default `min_chars` of 250. When `to_markdown` is false, `outcome()` line 502-503 (`markdown: if req.to_markdown { markdown } else { None }`) throws the whole thing away, and `text` is hardcoded `None`. This is not a rare path: `apps/extractor` (lib.rs:241-242) and `apps/plugin` (lib.rs:54-55) both build `FetchRequest::new(url)` with the default `to_markdown = false` and an `Auto`-family strategy — the bulk page-processing apps. Every page they fetch pays for a Markdown render that is dropped on the floor.
- **Root cause**: the char count and the Markdown output were fused into one call because they happened to share an implementation. The escalation decision needs a *predicate* ("is there at least `min_chars` of text?"), not a document; nothing forces it to materialize the whole conversion, and nothing forces it to keep going after char 250.
- **Impact**: one full DOM parse + markdown build + document-sized allocation per fetched page on the extractor/plugin hot paths, to answer a question that is settled by the first ~250 characters of text — and it's paid on cache hits too, since the tier logic runs after `HttpEngine::fetch` returns regardless of `resp.cache_hit`. The threshold passes for essentially every real page, so the work is discarded ~always. On a 500 KB page the walk + `String` growth + `normalize` pass is the majority of the conversion cost; the parse itself is the floor. Metric: time-in-`html_to_markdown` per fetch (extractor run), and allocated bytes per page.
- **Fix sketch**: split the predicate from the product. Add `markdown::text_len_at_least(html: &str, min: usize) -> bool` (or `text_len_capped(html, cap) -> usize`) that reuses the same `SKIP` list and `walk` shape but accumulates only a counter and returns as soon as `min` is reached — no output `String`, no `normalize`. In both tier arms, compute `markdown` only when `req.to_markdown` is actually set, and drive `enough` from the capped counter otherwise. `content_chars` in the `TierTrace` becomes `None` on that path (already `Option<usize>`, already documented at fetcher.rs:165-168 as "leaves it `None`" when not counted), so the trace contract holds. Trade-off: a second traversal helper to keep in sync with `html_to_markdown`'s skip rules — worth it on the bulk paths, and cheap to pin with a test asserting the two agree on the threshold for a fixture page.
