## 1. Let a query opt out of facets so it stops decoding 1000 stored docs to serve 20 hits

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: hot-path
- **File**: crates/core/src/search.rs:39-50,72-86 (trait/request shape); cost realized at crates/engine-search/src/lib.rs:255-299
- **Scenario**: Two hot paths. (a) `GET /search?q=вЂ¦&limit=20` вЂ” the default, user-facing. (b) `notify_saved_searches` (crates/server/src/worker.rs:399-416) runs one `Search::query` **per enabled saved search on every single job completion**, with `limit: 50`, and uses only the hit ids to claim never-before-seen matches вЂ” it never reads `SearchResponse::facets` at all.
- **Root cause**: `SearchRequest` has no facet toggle and `SearchResponse` unconditionally carries `SearchFacets`, so the trait contract obliges every implementation to compute facets on every call. `TantivyIndex::query` therefore does `sample_size = req.limit.max(FACET_SAMPLE)` (always в‰Ґ1000) and then, for **every** one of up to 1000 hits, calls `searcher.doc(*address)` вЂ” decompressing the stored doc including the full `body` вЂ” followed by `doc.to_json(&schema)` (serializes all stored fields, body included, into a `String`) and `serde_json::from_str` to parse it straight back. All of that, for `i >= req.limit`, exists solely to `+= 1` two `BTreeMap` counters (`app_counts`, `dataset_counts`). `app`/`dataset` are `STRING | STORED` with no `FAST` flag, so there is no columnar path to count them without touching the store.
- **Impact**: ~980 of 1000 stored-doc retrievals are pure waste on a default `limit=20` query. With ~20 KB bodies that is ~20 MB decompressed + ~20 MB of JSON string built + ~20 MB parsed **per query**, versus ~400 KB of useful work вЂ” roughly a 50Г— overread, all on a `spawn_blocking` thread. The saved-search path multiplies it by (enabled searches Г— job completions): 20 saved searches Г— 1000 jobs/day = 20k such queries/day whose facets are discarded.
- **Fix sketch**: Two layers. (1) Trait: add `pub facets: bool` to `SearchRequest` (`Default` в†’ `false`; `SearchRequest::new` sets it, the `/search` route sets `true`), and have `notify_saved_searches` leave it off вЂ” `sample_size` then collapses to `req.limit` (1000 в†’ 50) and `SnippetGenerator::create` can be skipped when nothing reads snippets. (2) Impl: add `FAST` to the `app`/`dataset` schema fields and count facets off the fast-field columns, so `searcher.doc()` runs only for `i < req.limit`; replace the `to_json` в†’ `from_str` round-trip with direct `doc.get_first(f.title).and_then(|v| v.as_str())` reads. The schema change is safe to ship вЂ” `TantivyIndex::new` already has a `body_is_stored` detect-and-rebuild-empty path (lib.rs:42-80) that is the exact precedent for a derived-index migration. Trade-off: `facets: false` defaulting on the struct means any new caller must opt in вЂ” acceptable, since the only facet consumer is the one HTTP route. **Metric**: count `searcher.doc()` calls per query and p95 `GET /search` latency at limit=20 over a в‰Ґ100k-doc index.

## 2. Give `HttpClient` a binary/streaming body path вЂ” a `String` body makes PDFs, ZIPs and spreadsheets unfetchable

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/core/src/engine.rs:161-171,288-292
- **Scenario**: Government and institutional sources вЂ” the platform's core domain вЂ” publish primary data as PDFs, ZIPs and XLSX. A user wants to fetch the CMS RVU release ZIP, a grants NOFO PDF, or a bulk XLSX and hand the bytes to an extractor/plugin. Today they cannot, at all, through any engine.
- **Root cause**: `HttpResponse.body` is typed `String`, and `engine-http`'s `read_body_capped` finishes with `String::from_utf8_lossy(&buf).into_owned()` (crates/engine-http/src/lib.rs:455-465). Any non-UTF-8 payload is silently mangled into replacement characters rather than erroring вЂ” the fetch "succeeds" and returns garbage. There is also no `content_type` on `HttpResponse`, so a caller cannot even detect that it got a binary and bail. The codebase already documents the gap as an accepted constraint it works around: `crates/apps/cms-fee-schedule/src/lib.rs:11-13` вЂ” *"It deliberately does NOT download the (binary, large) ZIP вЂ” the http engine yields a `String` body, so instead it reads the server-rendered RVU index page and detects the `rvuYYq` release tokens in it."* That app is reduced to a release **watcher** purely by this trait's type.
- **Impact**: Whole source classes are off-platform. `cms-fee-schedule` can only answer "is there a newer release?" and hands the actual ingest to an out-of-band `npm run ingest:pfs` script вЂ” the value stays outside pumper. Secondary cost: because the body is one in-memory `String`, `mpsv-vpm` (crates/apps/mpsv-vpm/src/lib.rs:128) materializes a **188 MB** download entirely in RAM, plus the transient `Vec<u8>` before `from_utf8_lossy` вЂ” ~2Г— peak per concurrent fetch of that job, with no streaming option. Distinct from the vision-scan item #156 (browser-engine *screenshot/PDF generation*): this is fetching binaries the origin already serves.
- **Fix sketch**: Additive, no break to existing callers. Add `pub content_type: Option<String>` to `HttpResponse` (already available in the headers loop at engine-http lib.rs:427). Then add `pub download_to: Option<PathBuf>` to `HttpRequest`: when set, `read_body_capped` streams chunks to that path (same `max_body_bytes` cap enforcement, same abort-on-overflow) and returns an `HttpResponse` with an empty `body`, so a 188 MB artifact never enters RAM. The artifacts root convention already exists (`data/artifacts/<app>/<job_id>/`, documented in docs/features/extraction.md source-mode). Guard the mangle-silently case: when `download_to` is `None` and `content_type` is non-text, return a typed error naming the content-type instead of lossy-decoding. Requires a `crates/core/src/fetcher.rs` passthrough for `Fetcher` callers. **Metric**: peak RSS during an `mpsv-vpm` run; `cms-fee-schedule` able to land the ZIP as an artifact.

## 3. Add `offset` + a true `total` to `SearchRequest`/`SearchResponse` вЂ” search is the one list surface with no page 2

- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/core/src/search.rs:39-56,72-76
- **Scenario**: A user searches a large corpus (e.g. `grants/unified`), gets 20 hits back with `count: 20`, and wants to (a) know how many opportunities actually matched and (b) look at results 21-40. Neither is possible. Their only lever is `limit`, clamped to 100 at crates/server/src/routes.rs:2033.
- **Root cause**: `SearchRequest` carries only `q`/`limit`/`app`/`dataset`/`fuzzy` вЂ” no `offset` or cursor вЂ” and `SearchResponse` is `{hits, facets}` with no match total. The `/search` route's `count` field (routes.rs:2041) is `results.hits.len()`, i.e. the **page size**, not the match count: it reports `count: 20` whether 20 or 50,000 documents matched, which reads as a total and is not one. The gap is a consistency outlier rather than a deliberate choice вЂ” a prior campaign shipped keyset cursors across every other list surface (jobs routes.rs:665-675, schedules :957-966, dataset records :1132-1139, changes :1386-1398, all via the shared `keyset_cursor` helper at :639), and search was simply never brought along. `docs/features/search.md` "Known gaps" lists semantic search and autocomplete but does not mention pagination, suggesting it was overlooked, not declined.
- **Impact**: Search caps out at 100 results per query with no way to go deeper вЂ” on a corpus where the facet sampler routinely sees 1000+ matches, the long tail is unreachable and relevance can't be judged against a denominator. Any client wanting page 2 must re-issue with a bigger `limit` and slice client-side, which re-runs the whole BM25 query. No API consumer can build a paged result list.
- **Fix sketch**: Add `pub offset: usize` (`Default` в†’ 0) to `SearchRequest` and `pub total: u64` to `SearchResponse`. In `TantivyIndex::query`, swap `TopDocs::with_limit(sample_size)` for `.and_offset(req.offset)` and get the exact total from a `Count` collector combined via `tantivy::collector::MultiCollector` вЂ” one collector pass, so the total is close to free relative to the store reads that dominate today (see finding #1; both changes land in the same `spawn_blocking` closure and should ship together). Then have the `/search` route expose `offset`, rename its misleading `count` to `total` fed from `SearchResponse::total`, and keep `hits.len()` implicit. Score-ordered offset pagination is correct here since the index is a derived artifact and the ranking is stable within a reader generation вЂ” a keyset cursor would need a tiebreak field and isn't worth it. Trade-off: deep offsets (>10k) get progressively costlier in Tantivy; clamp `offset` at the route the way `limit` already is. **Metric**: `GET /search?q=x&offset=100` returns distinct hits; `total` matches a direct `Count` over the same query.
