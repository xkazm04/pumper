## 1. Replace the per-key store reads in `role_trends` and ARES enrichment with two bulk reads

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: n-plus-one
- **File**: crates/apps/mpsv-vpm/src/lib.rs:296-328 (trends), crates/apps/mpsv-vpm/src/lib.rs:421-426 (ARES skip-existing)
- **Scenario**: Every daily 06:00 run, after aggregation, the app walks each national (`|ALL|`) agg cell and awaits `ctx.datasets.history(&ctx.app, "role_region_agg", key, 10)` — one SQLite round-trip per cell. It then walks every distinct employer IČO from this run's samples and awaits `ctx.datasets.get(&ctx.app, "employers", ico)` — another round-trip per IČO, purely to decide "already enriched, skip".
- **Root cause**: Both loops are written as point lookups inside a `for` over an in-memory collection, and both are `await`ed sequentially. `Datasets::history` (`crates/core/src/datasets.rs:293`) and `Datasets::get` (`:589`) each issue their own `sqlx` query against the pool; neither loop has a batch form. The store already exposes set-shaped reads — `changes_since(app, dataset, since, limit)` (`:316`) returns a dataset's revisions newest-first in one query, and `list(app, dataset, limit)` (`:605`) returns all records in one — but they are not used here.
- **Impact**: `|ALL|` cells are (distinct CZ-ISCO code × orgType) surviving `minCount`, realistically ~1.5–4k; distinct IČOs are bounded by `samplesPerGroup × unit groups` (~2k at defaults, since `distinct_icos` runs over `sample_items`). That is roughly **4–6k sequential round-trips per run** on top of `upsert_many`'s already-per-row writes, all serialized behind `await`. Collapsing to 2 queries removes essentially all of that latency; the trends loop is also the only reason the run touches `record_revisions` at all.
- **Fix sketch**: (a) Trends — one `ctx.datasets.changes_since(&ctx.app, Some("role_region_agg"), None, N)` before the loop, group the returned `Revision`s by `key` into a `HashMap<String, Vec<Revision>>` (they arrive newest-first, so per-key order is preserved and `revs.first()` / `.next_back()` semantics carry over unchanged), then drive the existing `count_of` / `prev` logic off the map. (b) ARES — one `ctx.datasets.list(&ctx.app, "employers", N)` into a `HashSet<String>` of keys, and replace the `ctx.datasets.get(...).is_some()` test with a set lookup. Trade-off: both bulk reads need a sane `limit`; pick one above the expected cardinality and log when it is hit, since a silent truncation would mis-classify cells as `"new"` or re-fetch known IČOs.

## 2. Drop the parsed feed after aggregation instead of pinning ~300k postings through the network phase

- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: memory-footprint
- **File**: crates/apps/mpsv-vpm/src/lib.rs:139-144, 177-237, 409-464
- **Scenario**: The 188 MB response body is correctly freed right after parse (`drop(resp)`, :142), but `feed` itself stays live for the rest of `run()` — through the trends history loop, the ISPV `list`, and the ARES enrichment phase, which performs up to `aresMaxLookups` (50) sequential HTTP fetches through the politeness governor. The whole typed corpus sits in RAM for the minutes that phase takes, while the worker may be running other apps concurrently.
- **Root cause**: `feed` is bound at function scope (:139) and its last read is the aggregation loop (:177-237); nothing drops it. Everything downstream works off the small derived collections (`agg_items`, `sample_items`, `gap_cells`, `posted_ages`). The existing `drop(resp)` shows the intent — it just was not extended to the parsed side, which is the larger and longer-lived of the two.
- **Impact**: `Vec<Posting>` over ~300k postings, each holding several `Option<String>` (title, dates, IČO, name, URL, skills `Vec<IdRef>`), is on the order of **100–200 MB held for the whole post-aggregation run** — the dominant residency of this app, and held across network waits rather than CPU work. Freeing it after the loop cuts steady-state footprint to the aggregates (a few MB). Note the ~370 MB *peak* during `serde_json::from_str` (source `String` + parsed `Feed` both live) is not addressable here: `HttpResponse.body` is a `String` (`crates/core/src/engine.rs:165`) with no streaming/reader surface, so a true fix needs an engine change — out of scope for this context.
- **Fix sketch**: Wrap the parse + aggregation loop in a block that returns the derived values (`cells`, `regions`, `groups`, `gap_cells`, `posted_ages`, `kept`, `filtered_old`, `ref_date`, `total`, `considered`), so `feed` drops at the block's end; or, minimally, `drop(feed)` immediately after the loop closes at :237 (`total`/`considered` are already captured as `usize` at :144-145). No behavior change, no readability cost.

## 3. Aggregate the skills and education already parsed off every posting into demand datasets

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/apps/mpsv-vpm/src/lib.rs:732 (`pozadovanaDovednost`), 724 (`minPozadovaneVzdelani`), 840-883 (`as_sample`), 230-236 (sample reservoir)
- **Scenario**: A user asks "which skills are actually in demand for CZ-ISCO 2512, and where?" or "what does a required degree add to the posted median?" — the two questions that turn a salary table into labour-market intelligence. Today neither is answerable: `role_region_agg` carries only counts and the salary distribution.
- **Root cause**: The app already deserializes `pozadovanaDovednost: Option<Vec<IdRef>>` and `minPozadovaneVzdelani: Option<IdRef>` for **every** posting, and the aggregation loop already visits every posting with `czisco`, `kraj`, `org` and `salary` in hand. But both fields are only read inside `as_sample`, which is called at most `gather_cap` times per unit group and then trimmed to `samplesPerGroup` (4). For every other posting the parsed skills and education are dropped on the floor at the end of the iteration. The gap exists because sampling was built for JD reference text, and the aggregate side was never extended to the non-salary dimensions.
- **Impact**: A whole product dimension is one `HashMap` away — **zero additional fetches, zero additional parsing, one extra pass-free `entry()` per posting** in a loop that already runs. It also unlocks the two named backlog items (vision-scan-2026-07-10 INDEX #150 "Skills-demand ranking per occupation", #215 "Education-level salary premium dimension") and feeds the change-detection substrate for free: once persisted, `role_trends`'s revision-history technique applies verbatim to give *rising/fading skills*, which is the sharper signal.
- **Fix sketch**: Two new accumulators alongside `cells`/`gap_cells`, filled in the existing loop at :192-236. (a) `skill_demand: HashMap<(String, String), Cell>` keyed `(unit_group(&czisco), skillId)` — iterate the posting's `pozadovanaDovednost` ids and `.add(salary)`, giving both a posting count and the salary distribution for postings demanding that skill; emit as `skill_demand` keyed `{unitGroup}|{skillId}` under the same `min_count` gate, with the per-group count as the denominator for a share. (b) `education_agg: HashMap<(String, String), Cell>` keyed `(unit_group(&czisco), educationId)` — reuse `Cell::to_value`'s percentile shape so the premium is a median-vs-median read against the group's `ALL` cell, never a fabricated delta. Both reuse `Cell`/`unit_group`/`min_count` and follow the established skip-thin-cells discipline. Break down as: skills first (bigger signal, and the trends technique composes onto it next run), education second. Note `IdRef` ids are codebook URIs (`"Dovednost/..."`) — key on them as opaque ids, do not substring-match, per the `typMzdy` lesson in harness-learnings.
