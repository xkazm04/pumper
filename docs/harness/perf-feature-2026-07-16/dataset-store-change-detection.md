## 1. Batch `upsert_many` into chunked write transactions instead of one BEGIN IMMEDIATE per record

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: write-contention / hot-path
- **File**: crates/core/src/datasets.rs:461-476 (`upsert_many`), :136-169 (`upsert`)
- **Scenario**: A `grants-gov` / `eu-sedia` / `census` run calls `ctx.upsert_many` (or `sync_many`, which delegates to it at app.rs:307) with the full listing вЂ” thousands of records in one batch. This is the single most-executed write path in the product: every job run, every schedule firing.
- **Root cause**: `upsert_many` is a plain `for` loop calling `self.upsert(...)` per item. Each `upsert` does its own `pool.acquire()` в†’ `BEGIN IMMEDIATE` в†’ SELECT + UPDATE/INSERT + `add_revision` в†’ `COMMIT`. So each record is an independent SQLite write transaction: its own WAL commit + fsync, its own acquire/release of the pool connection, and its own acquire/release of the database-wide write lock. The atomicity work that shipped (BEGIN IMMEDIATE around one record) is correct but was applied at the wrong granularity for batches вЂ” the transaction boundary is per-record where the caller's unit of work is per-batch.
- **Impact**: 5k-record run = 5k write transactions в‰€ 5k WAL commits/fsyncs + 5k pool acquires, where ~10 chunked transactions would do. On WAL+fsync the commit is the dominant per-record cost by far (typically ~0.5вЂ“2 ms each), so a batch that could finish in a few hundred ms takes seconds-to-tens-of-seconds. Second-order: the write lock is taken and released 5k times, so every other app's worker (concurrency > 1, `busy_timeout` 5 s) contends 5k times per batch instead of 10 вЂ” this is the mechanism behind cross-app write stalls during a large sync. Metric to watch: wall time of `upsert_many` per 1k records, and `pumper` job duration for the ingest apps before/after; expect ~10вЂ“50Г— on the write phase.
- **Fix sketch**: The seam already exists вЂ” `upsert_in_tx(conn, ...)` (:174) takes `&mut SqliteConnection` and does the whole per-record readв†’writeв†’`add_revision` sequence. Add `upsert_many` chunking: acquire one connection, and per chunk of ~500 items issue one `BEGIN IMMEDIATE`, loop `Self::upsert_in_tx(&mut conn, ...)` accumulating into `UpsertSummary`, then `COMMIT` (rolling the chunk back on error, exactly as `upsert` does at :164-167). Keep the single-record `upsert` as-is for its callers. Trade-offs to state in the doc: (a) a failure now rolls back its whole chunk rather than one record вЂ” chunking bounds the blast radius and keeps the summary honest if you only report committed chunks; (b) a long write transaction holds the write lock for its duration, so the chunk size is the knob trading throughput against other apps' worst-case wait (500 Г— ~0.1 ms of non-commit work stays well inside the 5 s `busy_timeout`).

## 2. Make `detect_removed` one transaction instead of 2 autocommit writes per removed key

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: write-contention / hot-path
- **File**: crates/core/src/datasets.rs:417-458
- **Scenario**: A full-snapshot `sync_many` where the upstream source drops a large slice of the dataset вЂ” a grants source retiring a closed program batch, a job board expiring a season's postings. `detect_removed` then tombstones hundreds-to-thousands of keys in one call.
- **Root cause**: Two independent problems from the same loop. (1) **Perf**: for each removed key the loop issues an `UPDATE records SET removed_at` and an `add_revision(&self.pool, ...)` вЂ” both against `&self.pool`, i.e. two *separate autocommit transactions* per key, each a WAL commit and a fresh write-lock acquisition. (2) **Consistency**: `upsert` was hardened to wrap its record-write + revision-append in one `BEGIN IMMEDIATE` precisely because a split there corrupts the revision chain the change feed relies on вЂ” `detect_removed` writes the same two rows and never received that treatment. If the process dies or the `add_revision` errors between the two statements, the record is tombstoned with no `removed` revision: the change feed and any dataset watch/trigger silently never fire for that removal, and no retry will re-emit it (the next sync sees `removed_at` already set and the key still absent, so the loop never revisits it вЂ” the signal is lost permanently, not merely delayed).
- **Impact**: 2 write transactions per removed key вЂ” a 2k-key removal is 4k commits where 1 would do, on top of the `upsert_many` cost from finding #1 in the same `sync_many` call (app.rs:307-312). Plus a durability hole in the removed-record signal, which is the differentiated product output ("delisted grants, closed vacancies") rather than an internal detail.
- **Fix sketch**: Acquire one connection, `BEGIN IMMEDIATE`, and run the loop's `UPDATE` + `Self::add_revision(&mut *conn, ...)` on it (`add_revision` is already generic over `E: SqliteExecutor`, so it needs no change), then `COMMIT` / `ROLLBACK` on the same pattern as `upsert` at :155-168. Chunk it like finding #1 if the removal set can be large. Keep the existing empty-`present` guard (:426-428) вЂ” that check is correct and load-bearing. Metric: commits per `sync_many` (expect `2 Г— removed` в†’ `~1`), and a test that kills the sequence mid-loop and asserts record-state and revision-count agree.

## 3. Add a record/revision retention and delete API вЂ” the store can only grow

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap / unbounded-growth
- **File**: crates/core/src/datasets.rs:123-762 (whole `Datasets` impl вЂ” no delete/prune exists), :260-290 (`add_revision`), :425-427 (the comment referencing a delete path that isn't there)
- **Scenario**: A user points pumper at a source and puts it on a 6-hourly schedule вЂ” the product's core promise, "turning one-off scrapes into datasets that accrue over time" (module doc, :1-5). A year later they want to bound the DB, drop a dataset they no longer track, or honor a removal request for a scraped record. There is no way to do any of it short of hand-editing SQLite.
- **Root cause**: `Datasets` exposes 16 public methods вЂ” every one is a read or an upsert. There is no `delete_record`, no `delete_dataset`, no `prune_revisions`. `record_revisions` stores a **full JSON snapshot per revision** (`data TEXT`, migration 0005) and is append-only forever. The gap is visible in the code's own text: `detect_removed`'s comment says "callers that legitimately empty a dataset should delete explicitly" (:425-427) вЂ” pointing at an API that was never written. The platform already has the pattern to copy: `Cache::purge_expired` (cache.rs:130) plus a janitor loop wired in server `main.rs:167`; the dataset store just never got its equivalent. The only `DELETE /вЂ¦/datasets/вЂ¦` route (routes.rs:2857) drops the *search index*, not the records.
- **Impact**: Unbounded, monotonic growth in normal operation. A 10k-record dataset on a 6-hourly schedule with 10% churn в‰€ 1k full snapshots/day, forever вЂ” roughly GB/year per active dataset at a few KB/record, with no ceiling and no operator control. It degrades the `idx_revisions_feed` index, the change-feed and export paths, and every backup, and it is a real blocker for GDPR-style deletion on scraped personal-ish data (`mpsv-vpm` vacancies, `cz-labour`). Product-wise it also blocks the obvious adjacent features: "delete this dataset", "keep 90 days of history", "reset and re-scrape".
- **Fix sketch**: Break it into three shippable steps rather than one retention subsystem. (a) **Delete API**: `Datasets::delete_record(app, dataset, key)` and `delete_dataset(app, dataset)`, each removing `records` + `record_revisions` rows in one `BEGIN IMMEDIATE` (same pattern as `upsert`), surfaced as `DELETE /datasets/{app}/{ds}` and `DELETE /datasets/{app}/{ds}/records/{key}`. This alone closes the honesty gap at :425-427 and the deletion-request need. (b) **Revision pruning**: `prune_revisions(older_than: DateTime<Utc>, keep_min_per_key: i64) -> Result<u64>` вЂ” delete by `created_at` while always keeping the latest N per key so the diff chain and `history` stay usable; the `(app, dataset, key, revision)` PK and `idx_revisions_feed` already support both predicates. (c) **Janitor**: call it from the existing `main.rs` janitor tick alongside `cache.purge_expired`, driven by a `StorageConfig` retention setting (default off вЂ” silently deleting a user's accrued history must be opt-in). Metric: `record_revisions` row count and DB file size per dataset over a week; the win is turning an unbounded curve into a flat one.
