## 1. Make `data-sources.toml` load-bearing — today it is inert prose that has already drifted from the registry

- **Severity**: High
- **Lens**: feature-scout
- **Category**: inert-data
- **File**: catalog/data-sources.toml:1-27, catalog/README.md:38-70
- **Scenario**: `ONBOARDING.md` §10 and `catalog/README.md:3-7` declare the TOML "the authoritative, machine-readable list of every data pipeline… LLMs parse the TOML", and Path B makes updating it a contract. But no Rust code parses it. A grep across the workspace finds `catalog/data-sources.toml` only inside `//!` doc comments (`grants-gov`, `eu-sedia`, `ca-grants`, `mpsv-vpm`, `mpsv-ispv`, and the two `census-*` "deliberately not listed" notes). The only catalog file any code reads is `connector-docs.json`, via `connector-api-watch` (`crates/apps/connector-api-watch/src/lib.rs:34,104-107`). So the file that is supposed to answer "what does this machine scrape" is a hand-maintained doc with no reader and no gate — and it has drifted:
  - `ca-grants` has `cron = ""` (line 122) but `ca-grants/src/lib.rs:39-41` returns `Some("0 30 9 * * *")`; same for `eu-sedia` `cron = ""` (line 237) vs `eu-sedia/src/lib.rs:45-47` `Some("0 0 10 * * *")`. The catalog says "not on the scheduler" about two apps that are.
  - The README's "rendered from the TOML" snapshot (lines 42-68) has 23 rows and claims "**5 live** · 17 planned · 1 blocked"; the TOML has 24 sources and 6 live — `connector-api-watch` (lines 409-423, `status = "live"`) is missing from the table entirely.
  - Nine registered apps that ingest real external data are absent from the "every data pipeline" list: `cms-fee-schedule`, `trade-wages`, `valuation-multiples`, `state-tax`, `homewyse-pricing`, `crawl`, `watch`, `hackernews`, `research`. Only `census-density`/`census-nonemp` document *why* they're excluded.
- **Root cause**: The catalog was designed as a human/LLM-readable artifact, and its consistency with `AppRegistry` + `ScrapeApp::schedule()` is enforced only by author discipline. There are three copies of the same truth (TOML, README table, Rust) and zero of them check each other. The doc-sync Stop hook explicitly *skips* the catalog (`scripts/docs/check-doc-sync.mjs:30` — `/^catalog\//`), so even the one automation that could have caught drift is opted out.
- **Impact**: Every consumer of the catalog — an operator, an LLM agent doing source-scout work, or the downstream grant-writing app — gets answers that are wrong today, silently. Drift is unbounded: it grows with every app added, and 9/24 apps are already invisible. The claimed "uniform, greppable answer to what data do we have, how fresh, how trustworthy" does not hold.
- **Fix sketch**: Add `pumper_core::catalog` — `#[derive(Deserialize)] struct Source { id, app, market, name, url, category, engine, access, cadence, cron, status, confidence, dataset, notes }` + `struct Catalog { source: Vec<Source> }` and `Catalog::load()` mirroring `Config::load()` (env override `PUMPER_CATALOG`, default `catalog/data-sources.toml`). Then make it enforceable and useful:
  1. A `#[test]` in the server crate that cross-checks the parsed catalog against `AppRegistry`: every `status = "live"` entry must have a non-empty `app` that is registered; its `cron` must equal that app's `ScrapeApp::schedule()` (both directions, so `""` vs `Some(...)` fails); every registered non-exempt app must appear (with an explicit `exempt` allow-list for `census-*`, `crawl`, `extractor`, `plugin`, `readable`, `research`, `watch` rather than today's silent omission). This alone catches all four drifts above.
  2. `GET /catalog/sources` on the router (`crates/server/src/routes.rs`), with `?market=`/`?status=`/`?category=` filters — the catalog becomes an API a downstream app can query for "which markets are launch-grade" instead of scraping a TOML out of a sibling repo.
  3. Generate the README snapshot table from the parsed TOML in the same test (assert-or-write), killing the third copy.
  Load cost is trivial and one-shot (14 KB at boot, same shape as `Config::load()` at `crates/core/src/config.rs:58-72`), so this is a correctness win with no perf trade-off.

## 2. Stop imposing one source's 300s timeout on every request in the fleet — use the per-request override that already exists

- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: config
- **File**: config.toml:21-27, crates/core/src/engine.rs:121-125
- **Scenario**: `config.toml:23-26` documents its own reasoning: "Total per-request timeout. Raised from the 30s default so bulk open-data downloads (e.g. mpsv-vpm's ~188 MB national vacancy feed) complete on typical links". So `[http] timeout_secs = 300` — a 10× increase over `HttpConfig::default()`'s `timeout_secs: 30` (`crates/core/src/config.rs:219`) — applies to **every** HTTP request from **every** app, because one daily source needs it. Meanwhile `HttpRequest.timeout_secs: Option<u64>` (`crates/core/src/engine.rs:121-125`) already exists and is documented as "Overrides the client-global `[http] timeout_secs` for this request only" — and `mpsv-vpm` does not use it: `crates/apps/mpsv-vpm/src/lib.rs:129` builds a plain `HttpRequest::get(&url)`, and its own module doc (line 27) concedes "The full file needs a raised `[http] timeout_secs`". The mechanism to scope this was built and left unwired for the one caller that motivated the global change.
- **Root cause**: The global knob was the fastest way to unblock the 188 MB `mpsv-vpm` fetch; the per-request override landed separately and the config was never walked back. Nothing pins the two together, and `config.toml` is not covered by tests.
- **Impact**: A hung or blackholed host now occupies a worker slot for 300s per attempt instead of 30s. With `[worker] concurrency = 4` and `[http] retries = 3`, a single dead endpoint burns a slot up to `job_timeout_secs = 900` (config.toml:10) — the job timeout, not the HTTP timeout, becomes the real backstop, i.e. ~15 min held instead of ~2 min. Four such jobs (e.g. the four `browser`/`scrape` state sources once built, or any of the `claude`-engine apps' web fetches) stall the entire fleet for 15 minutes, and the scheduler's 24 catalog cadences queue up behind them. Failure detection latency is 10× worse fleet-wide to serve one source's daily run.
- **Fix sketch**: Set `HttpRequest.timeout_secs = Some(300)` on the bulk fetch in `crates/apps/mpsv-vpm/src/lib.rs:129` (`let mut req = HttpRequest::get(&url); req.timeout_secs = Some(300);`), then restore `config.toml:26` to `timeout_secs = 30` (matching the documented default) and replace the comment with a pointer to the per-request override as the pattern for bulk sources. Do the same for any future `engine = "bulk"` catalog entry (`grants-gov-xml`, `irs-eo-bmf`, `au-acnc`, `uk-360giving`, `cra-charities` — all `planned` today), so "bulk needs a long timeout" is expressed per-request rather than by degrading the global. Metric to watch: p99 time-to-failure for a non-responding host, and worker-slot occupancy (`heartbeat_secs = 30` already gives the reaper the signal to measure against). Trade-off: each bulk app must remember the override — the catalog test from finding #1 can assert that every `engine = "bulk"`/188MB-class source's app sets one.

## 3. Turn `cadence` + `dataset` into an actual freshness monitor — the catalog asserts liveness nothing verifies

- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: automation
- **File**: catalog/data-sources.toml:20-26, catalog/data-sources.toml:31-45
- **Scenario**: Each `[[source]]` already carries the three fields a staleness check needs: `status` (`live`), `cadence` (`daily`/`weekly`/`monthly`/`quarterly`/`annual`), and `dataset` (the `Datasets` dataset it writes via `ctx.upsert` — `opportunities`, `orgs`, `awards`, `role_region_agg`, `wages`, `connector_docs`). Nothing joins them. `grants-gov` claims `cadence = "daily"`, `status = "live"`, `dataset = "opportunities"` and its `notes` cite a verification from a point in time ("Verified live: 1366 posted opps"); `ca-grants` and `mpsv-vpm` say "Verified live 2026-07-05". If Grants.gov changes its POST contract tomorrow and the app starts failing, the catalog keeps asserting `live`/`confidence = 5` forever — the `notes` are a snapshot of a manual check, not a live signal. The product's whole premise is freshness ("how fresh, how trustworthy" — `catalog/README.md:11`), and freshness is the one thing it can't answer about itself.
- **Root cause**: The catalog was written as documentation, so its most machine-actionable fields (`cadence`, `dataset`, `status`) were never given a runtime consumer. The pieces to build on all exist independently — `Datasets` records carry timestamps, the scheduler already ticks every `schedule_tick_secs = 15` (config.toml:13), and `[webhooks] failure_url` (`crates/core/src/config.rs:29-38`) is a working cross-app "a job failed" firehose. The gap is that a *silent* pipeline (jobs succeeding but writing nothing, or a source quietly delisted) fires no `job.failed` and is therefore invisible.
- **Impact**: Silent data-staleness on any of the 6 live sources goes undetected until a human re-reads the catalog and manually re-verifies — the mode the `notes` timestamps prove is in use today. Scales badly: the catalog is explicitly a roadmap ("18 planned"), so the manual-verification burden grows ~4× as the planned sources land.
- **Fix sketch**: Build on the `pumper_core::catalog` loader from finding #1. Add `cadence_secs()` on `Source` mapping the enum to a max-expected-age (`daily` → 24h, `weekly` → 7d, `monthly` → 31d, `quarterly` → 93d, `annual` → 366d; `on-demand`/`one-time` → exempt), apply a grace multiplier (~2×) to tolerate one missed run, and expose `GET /catalog/health` returning per-source `{ id, app, status, cadence, dataset, last_write_at, age_secs, stale: bool }` by querying the newest `updated_at` in each entry's `dataset` scoped to its `app`. Then wire the automation: a scheduled sweep that emits a `source.stale` event through the existing `[webhooks] failure_url` path when a `status = "live"` source's dataset hasn't been written within its cadence window. This makes `confidence` and `status` self-checking rather than aspirational, and gives the planned-source backlog a real definition of done. Natural follow-on: `notes`' hand-written "Verified live YYYY-MM-DD" strings get replaced by the endpoint's `last_write_at`.
