## 1. Gate the metered research call on vintage freshness, not just the 24h request cache

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: research-cost
- **File**: crates/apps/state-tax/src/lib.rs:95-103
- **Scenario**: An operator (or a `POST /schedules` cron on any of the four apps) refreshes `state-tax` for `year=2025` on Monday, then again on Wednesday. The Wednesday run pays the full agentic price — a 30-turn, web-searching Claude run over 51 jurisdictions — to re-derive tax-year-2025 constants that were fixed when the IRS/state legislatures published them and cannot have changed. Same story for `trade-wages` (a frozen BLS OEWS vintage) at `max_turns` 25.
- **Root cause**: Each app's `run()` goes straight from param parsing to `trades_common::research_json(&ctx, …, request)` (state-tax:102, trade-wages:97, valuation-multiples:90, homewyse-pricing:99) with no check of what it already holds. The only spend guard beneath it is `AppContext::research`'s cache (`crates/core/src/app.rs:235-254`), which is keyed on the *request* and TTL-bound at 86400s by default. That is calibrated for volatile scraping: it makes an identical re-run free for 24h, then lets the next one re-pay 100%. But the freshness contract of these four datasets is the `year` param, not wall-clock — and nothing connects the two. The apps already store `year` on every record (state-tax:149, trade-wages:162, valuation-multiples:140), so the information needed to skip is present and unused.
- **Impact**: A no-op refresh costs the app's entire run cost. These are the fleet's only `ctx.research` callers and the priciest path in the product (20-30 turns + web search + page fetches each; ~4 full runs to refresh the domain). Any cadence tighter than the data's real cycle — annual for state-tax/trade-wages, quarterly at most for valuation-multiples — is pure burn, and each redundant run also costs minutes of latency and a worker slot. A daily schedule on all four would spend ~365× what the annual facts warrant.
- **Fix sketch**: Before building the `ResearchRequest`, read back the app's own dataset and short-circuit. In `state-tax`, `ctx.datasets.get("state-tax", "tax", "federal:US")` → if `rec.data["year"] == year` and no `force` param, return the existing counts with `"skipped": "vintage already held"` and `cost_usd: 0` without calling `research_json`. Same shape in `trade-wages` (`get("trade-wages", "wages", "US:Plumbing")`). Trade-off worth respecting per app: tax-year and OEWS-vintage facts are genuinely frozen, so vintage-equality is the right gate; `homewyse-pricing` and `valuation-multiples` track figures that really do drift within a year, so gate those on record age instead (a `max_age_days` param, default ~90) rather than on `year` alone. `force: true` always re-runs. This belongs in the apps, not the core cache — the core cache is request-identity, this is domain freshness.

## 2. Give operator_economics a state dimension — the tax data is already per-state and the join throws it away

- **Severity**: High
- **Lens**: feature-scout
- **Category**: feature-gap
- **File**: crates/apps/homewyse-pricing/src/lib.rs:49-61,130
- **Scenario**: A Ledgerline plumber in Texas asks the two questions this dataset exists to answer — "what should I pay my first hire?" and "what should I charge for a drain clear?" — and gets a national number. Meanwhile the operator's *tax* set-aside, the one read that state-tax was built to ground (state-tax:9-10), resolves to `illustrative_state_top_marginal_rate_median` (trades-common:502): the median rate across all 51 jurisdictions. A Texan (0%) and a Californian (13.3%) are handed the same middle number, which is right for neither.
- **Root cause**: The domain was built national-first and the join hardened that in. `trade-wages` and `valuation-multiples` key on `US:{trade}` and stamp `state = "US"` (trade-wages:125,161; valuation-multiples:114,139); trade-wages' own header already flags the intent ("per-state wage detail can layer on later", :16-17). `state-tax` *does* hold all 51 states (:129-153), but `unified::sync_operator_economics` iterates only `Trade::ALL` and emits `US:{label}` (trades-common:436-471), reducing the 51 real rates to one median (trades-common:423-430). Worse, `homewyse-pricing` already has the state axis half-built — `locality` is a param that flows into the record key (`{locality}:{trade}:{job}`, :130) — but `summarize_pricing` filters on `trade` alone (trades-common:518-535). So the moment anyone runs it for two localities, Texas and national prices are silently averaged into one envelope. The half-implemented axis doesn't just fail to help; it corrupts the unified row.
- **Impact**: This is the product's largest gap: the state read is what makes the data actionable, and it is the natural next surface rather than a new one. Cost of the extension is bounded because the axis is asymmetric — state-tax is *already* complete (zero extra research), pricing needs the locality param actually driven, and only wages needs new per-state OEWS research. Also a live scale bug: `list("homewyse-pricing", "pricing", 1000)` (trades-common:433) caps at 1000, and 51 localities × 5 trades × 4 jobs = 1020 rows silently truncates the summary.
- **Fix sketch**: Phase it. (a) Fix the contamination first — have `summarize_pricing` take a `locality` and filter on `r.data["locality"]`, and raise/paginate the 1000 cap. (b) Re-key the unified row `{state}:{trade}` and join the *real* `state:{ST}` tax record per state instead of the median, keeping `US:{trade}` as the national roll-up — pure win, no new research spend. (c) Then add `state` to `trade-wages` (OEWS publishes state-level occupation wages; batch several states per call to bound cost, and reuse `Trade::soc_code()` rather than re-asking the model to map SOC codes it already knows). Valuation multiples can stay national — broker comps are too thin per state to be honest at that grain, and saying so is better than fabricating it.

## 3. Key the trades datasets by vintage so year-over-year becomes a product

- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: data-freshness
- **File**: crates/apps/trade-wages/src/lib.rs:125,162
- **Scenario**: An operator wants the read that reference data uniquely enables: "plumber median wage is up 4.2% YoY" or "your state's top rate changed for 2026". Nothing can answer it. Worse, running `trade-wages` with `year=2023` after a `year=2024` run overwrites the 2024 band in place, and change detection reports the vintage rollback as a genuine `changed` revision — indistinguishable from a real wage move.
- **Root cause**: `year` is a first-class param and is stored on the record (`rec["year"] = json!(year)`, trade-wages:162) but is absent from every key: `US:{trade}` (trade-wages:125, valuation-multiples:114), `{locality}:{trade}:{job}` (homewyse-pricing:130), `state:{st}` (state-tax:151). So the datasets are a single latest-vintage snapshot, and two vintages collide on one key. `state-tax` compounds it by using `sync_many` (:170) — correct for a full 51-jurisdiction snapshot, but it means an older-year run replaces the whole current set. History survives only in dataset revisions, which is a debugging surface, not a queryable trend one.
- **Impact**: Bounded — today's national-latest read is still correct, and revisions mean nothing is truly lost. But it blocks a whole class of feature for free data (every past run already paid for its vintage), and it's the same shape as the roadmap's acknowledged `Census YoY trend layer awaits multi-vintage accumulation` gap (docs/features/apps.md:53). Fixing it now means vintages accumulate from the next run; leaving it means the trend layer's clock never starts.
- **Fix sketch**: Put the vintage in the key — `US:{year}:{trade}` for wages/valuation, `state:{year}:{ST}` for tax, `{locality}:{year}:{trade}:{job}` for pricing. Note the one-time cost, exactly as the taxonomy re-key did (docs/features/apps.md:34): existing rows orphan under their old keys and the next run repopulates. Keep `unified::sync_operator_economics` pointed at the latest vintage per trade (pick max `year` when listing) so the current read is unchanged, then add `wage_growth_yoy` / `rate_change` to the unified row once two vintages exist. `state-tax` must move `sync_many` → `upsert_many` under vintage keys, or its snapshot semantics will delete last year's set — the full-snapshot removal detection is only correct *within* a vintage.
