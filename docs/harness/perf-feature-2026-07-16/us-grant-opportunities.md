## 1. Enrich federal opportunities via `fetchOpportunity`, gated on the upsert summary

- **Severity**: High
- **Lens**: feature-scout
- **Category**: coverage-gap
- **File**: crates/apps/grants-gov/src/lib.rs:176-180 (normalize call) → crates/apps/grants-common/src/lib.rs:74-99 (`normalize_grants_gov`)
- **Scenario**: A user calls `GET /grants?min_award=250000` to find grants worth chasing. Every federal opportunity — the larger half of `grants/unified` — is silently excluded, because its `award_floor`/`award_ceiling`/`total_funding` are all null. Same for `eligibilities[]`: the app's own description advertises an `eligibilities` param (`12|13|25|99` for nonprofits), so a user *can* filter server-side, but the stored row never records which eligibility applied — the filter is unexplainable and un-auditable after the fact.
- **Root cause**: Search2's `oppHits[]` carries no money and no per-opportunity category/eligibility facets (those are search *filters*, not hit fields) — the code comments at `grants-common/src/lib.rs:84-90` state this correctly. So `normalize_grants_gov` hard-codes `award_floor`/`award_ceiling`/`total_funding` to `Value::Null` and `categories`/`eligibilities` to `[]`. The gap is honest but permanent: nothing in the pipeline ever visits the endpoint that *does* publish these fields (`POST /v1/api/fetchOpportunity`, `{"opportunityId": <id>}` → synopsis with `awardFloor`/`awardCeiling`/`estimatedTotalProgramFunding`/`applicantTypes`/`fundingActivityCategories`/`description`). `apps.md` already concedes the consequence ("`min_award` narrows to `ca-grants` alone") but frames it as an upstream limit rather than an unfetched endpoint.
- **Impact**: The single most decision-relevant field on a grant (how much money) is absent for ~100% of the federal corpus, making `GET /grants?min_award=` a CA-only filter and the cross-source `grants/unified` schema asymmetric in exactly the column users sort on. Aligns with existing backlog idea `2e82222c` ("Award-amount extraction and range filtering") but supplies the mechanism that title lacks. The naive implementation is an N+1 trap — one POST per opportunity × ~2,500 = 2,500 daily round-trips — which is presumably why it hasn't been done.
- **Fix sketch**: The N+1 is avoidable using state the run *already computes*. `ctx.upsert_many("opportunities", &items)` at `grants-gov/src/lib.rs:172` returns an `UpsertSummary` whose `new` + `changed` key sets are, in steady state, tens of rows — not thousands. Enrich only those: after the upsert, for each key in `summary.new ∪ summary.changed`, `ctx.engines.http.fetch(fetch_opportunity_request(id))` (reuse the `search2_request` builder shape at :250 — same POST+JSON headers, different URL), merge the synopsis fields into that hit before `normalize_grants_gov`. Add a `maxEnrich` param (default ~200, clamped) so the *first* backfill run — where every row is `new` — spreads across days instead of firing 2,500 requests; the fetcher's cache (`no_cache: false`) makes re-enrichment of an unchanged opportunity free anyway. Extend `normalize_grants_gov` to read the merged fields via the existing `money_scalar`/`money_range`/`str_list` helpers rather than hard-coding nulls. Trade-off: closing the money gap re-writes every federal unified row as `changed` on the first run after deploy — the same one-time cost `apps.md` already documents for the CA money fields.

## 2. Raise the default `rows` — 100 costs 25 round-trips and silently caps the federal corpus at 2,500

- **Severity**: High
- **Lens**: perf-optimizer
- **Category**: pagination-inefficiency
- **File**: crates/apps/grants-gov/src/lib.rs:46-48 (`default_params`), :69-80 (clamps), :139 (loop break)
- **Scenario**: The scheduled 09:00 UTC daily sync runs with `default_params` — `{rows: 100, maxPages: 25}` — which the doc comment at :40-41 asserts "are sufficient". Two things follow that the code contradicts: the run issues 25 sequential HTTP round-trips, and it stops at exactly 2,500 records no matter how many exist.
- **Root cause**: `rows` is clamped to `1..=1000` (:74) but defaults to 100 — a 10× under-use of the API's own page size. The loop is strictly sequential (each `fetch` awaits before the next `start += rows`), so page size maps 1:1 onto latency. Separately, the break condition `pages >= max_pages` (:139) is a *silent* terminator: it is indistinguishable in the result JSON from the legitimate `got < rows` / `start >= hit_count` exits. Note the asymmetry with the file's own honesty stance — the run *hard-fails* on the drift case (`hitCount > 0 && hits.is_empty()`, :148-153) but returns `Ok` after truncating an arbitrary fraction of the corpus.
- **Impact**: `rows: 1000` fetches the same 2,500 records in **3 round-trips instead of 25** (~88% fewer requests, and the run's wall-clock is round-trip-bound), and raises the ceiling from 2,500 to 25,000 at the *same* `maxPages: 25` — one default change buys both. The ceiling matters: federal-wide `posted|forecasted` plausibly exceeds 2,500, and any opportunity past the cutoff is never fetched, so it is invisible to `GET /grants`, `GET /grants/closing-soon`, the `closingSoon` digest, the search index, and cross-source dedup. It is not data loss (the run is upsert-only, so a truncated row goes stale rather than being swept — `sweep_closed` only flips *past-due* rows), but it is unreported coverage loss. The result *does* carry both `hitCount` and `fetched` (:198-199), so the truncation is derivable by a careful reader — it is simply never flagged.
- **Fix sketch**: Change `default_params` (:47) to `{"oppStatuses": "posted|forecasted", "rows": 1000, "maxPages": 25}`. Then make the cap honest: when the loop exits via `pages >= max_pages` while `start < hit_count`, push a string onto the result's `warnings` array — the `UnifiedOutcome.warnings` channel already exists and `merge_into` (`grants-common/src/lib.rs:41`) already surfaces it, so this needs no new result field; alternatively add a `truncated: bool` beside `pages`. Trade-off: 1,000-record pages mean larger response bodies and a coarser retry granularity on a mid-pagination failure — set `max_body_bytes` on the request builder (:254, currently `None`) if that becomes a concern. Apply the identical change to `ca-grants` (`default_params` :44, `limit` clamped `1..=1000` at :59, same silent `pages >= max_pages` break at :122).

## 3. Run the corpus-global unified tail once per sync, not once per source

- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: redundant-work
- **File**: crates/apps/grants-gov/src/lib.rs:180 and crates/apps/ca-grants/src/lib.rs:154 (both call `grants_common::finalize_unified`)
- **Scenario**: `grants-gov` runs at 09:00 UTC and `ca-grants` at 09:30 (schedules at `grants-gov:42` / `ca-grants:39`, deliberately offset "to spread the load"). Each calls `finalize_unified`, whose tail — `sweep_closed` then `link_duplicates` — operates on the **entire** `grants/unified` corpus, not on the batch the caller just fetched. So the whole cross-source corpus is swept and dup-linked twice every morning. The 09:30 sweep is near-guaranteed to find nothing: 09:00 already flipped every row whose `close_date < today`, and "today" has not changed in the intervening 30 minutes.
- **Root cause**: `finalize_unified` (`grants-common/src/lib.rs:59-70`) mixes one batch-scoped step (`sync_unified`, correctly takes `unified_items`) with two corpus-global steps that take only `ctx`. Because it is invoked per-source, the global steps' cost scales with **sources × corpus** rather than corpus. The mechanism is concrete: `sweep_closed` calls `ctx.datasets.list(UNIFIED_APP, UNIFIED_DATASET, 1_000_000)` (:159), which is `SELECT key, data, … LIMIT 1000000` — it deserializes every row's `data` JSON into a `Record`, including tombstoned rows (skipped only *after* loading, :162) and already-`closed` rows (the overwhelming majority over time, and never removable since `removed_at` is reserved for full-snapshot sources). The comment at :158 — "Local datasets are small (both sources cap well under this); one read" — is true today but is an assumption that decays monotonically, because the corpus only ever grows.
- **Impact**: Exactly 2× the necessary corpus-global work per day, by construction and independent of scale — plus 2× the JSON deserialization of the full corpus into memory. At today's few-thousand rows this is small (hundreds of ms, single-digit MB); the finding is the scaling shape, not a present outage. It compounds with source count: a third wired source makes it 3×. Distinct from the already-triaged `duplicate_pairs unbounded scan` (2026-07-14, Wave 8), which concerns the O(n²) inside `duplicate_pairs`; this is about how many times that scan is invoked, and would remain 2× even after that scan is bounded.
- **Fix sketch**: Split the tail. Keep `finalize_unified` doing the batch-scoped `sync_unified` + `drift_warnings` per source, and hoist the corpus-global `sweep_closed` + `link_duplicates` into a step that runs once per sync cycle. Lowest-friction option given the existing architecture: expose them as a small `grants` maintenance app on its own schedule (e.g. 09:45, after both sources land) and have `UnifiedOutcome` report `swept`/`cross_source_dups` from there — the `merge_into` shape (:34-49) already tolerates this since both fields are plain counts. If keeping the per-source call is preferred for result-shape stability, make the work idempotent-cheap instead: give `sweep_closed` a `list_filtered` predicate (`status IN (open, forecasted)` via the existing `JsonFilter::Eq` path at `core/src/datasets.rs:656`) so it loads only sweep candidates rather than the whole corpus. Trade-off: the maintenance-app split means a source's own result no longer reflects the sweep it triggered — counts move to a separate job's result, which is a small observability regression against a structural win.
